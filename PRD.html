Executive Summary

Canopy OS is a tree-native collaboration surface for humans and AI agents. It replaces the linear chat log with a spatial, navigable conversation tree where every message can branch, merge, and be referenced across contexts.

Think Hermes + Obsidian had a baby, and it was your chat app. The files ARE the knowledge base. The agent's memory IS the tree. The interface IS the memory — they're the same thing.

Canopy is not another chat app. It's a web OS with a chat window — dynamic micro-apps (cards) surface as you work, each with its own database, UI, and state machine. The agent designs the interface it wants to use to communicate with you.

Problem

Every existing AI chat interface — ChatGPT, Claude, Copilot, Gemini — is built on the same foundation: a linear scroll log designed for human-to-human conversation.

Solution: The Tree-Native Interface

Canopy replaces the scroll log with a spatial conversation tree. Every message is a node. Nodes can branch, merge, be referenced, and carry state. The tree IS the agent's memory — visible, navigable, and shareable.

🌳 Three-Level Navigation

Macro View — full conversation forest at a glance. Branch View — drill into any thread, see full context chain. Merge View — side-by-side comparison of alternative paths. Fluid transitions between all three levels. Keyboard-driven: j/k navigate, h/l drill in/out, m merge selected.

🔀 Branch, Don't Derail

Reply to ANY message in the tree — not just the last one. Replying to the latest extends the current branch. Replying to an older message creates a fork. The main thread stays clean. The agent understands branch context — it knows which path you're on and what led here.

📎 Multi-Message Reference

Reply to multiple messages at once. Select N messages → reply → new node with N parent edges. Colored edges show which context came from where. "Based on your database schema work from Monday AND the API concerns from Tuesday, here's the plan..."

🧵 Topics with #References

Auto-topic detection: the agent notices topic shifts and offers to create new topics. Slack-style #references between topics. Search sidebar: type, see 3 matches, one click injects that topic's entire tree into context — no need to open and read it. Topics have lifecycle: active → archived → deleted. Merge, split, cross-reference at will.

Card Ecosystem — "A Web OS with a Chat Window"

Canopy is an interface that has interfaces. It's its own OS. Cards are micro-apps that ship with their own HTML/JS/CSS, their own DuckDB-backed JSONL database, their own REST API, and their own state machine.

🗄️ Database-per-Card

Each card type gets its own storage: ~/.hermes/canopy/cards/{type}/events.jsonl. Git-friendly, diffable, append-only JSONL files. DuckDB queries them directly — no import step, no binary blob. pgREST-style fixed API: GET /api/cards/{type}/{id} — neither the model nor the card JS ever writes raw SQL.

🧩 Cards as Micro-Frontends

Each card ships HTML/JS/CSS (Web Components with shadow DOM). Sandboxed, auth-gated. Queries its own database via REST. Runs its own application logic. Has a state machine: sending → pending → bench → active → archived. Cards can be replied to — they're first-class nodes in the conversation tree.

🔄 Dynamic Thinking Interface (Iteration Cards)

While the agent is working — searching the web, executing code, reading files — the interface is dynamic. Open a sidebar, say "show me the searches," and a card pops up with live results. Highlight specific results, give feedback ("focus on this one"), and it goes BACK to the model mid-process. Each card is a bidirectional channel between you and the agent's thinking.

💬 Message-to-Card

Take any conversation and say "turn this into a card." The agent extracts structured data and renders a persistent, interactive view. 15 messages about a database schema → a Database Schema card with tables, columns, and relationships. Talk TO the card to modify it. Cards reference other cards via #links.

🧠 Skills Are Cards

Canopy cards ARE Hermes skills — rendered as UI. A skill that was a text instruction becomes an interactive app with its own database, API, and state. The skill system and the card system are the same thing. This is "a wild extension bolted onto Hermes."

Multi-User & Federation

👥 Approval Gates

Friends join threads. The agent listens to everyone but only acts on owner-approved input. "Delete the production database" → agent tags it "unapproved — waiting for owner." Approval granularity: per-message, per-thread, per-user. Side panel preserves full context — you always see what you approved and why. Immutable audit trail.

🤖 Multi-Profile

Different Hermes profiles participate in the same tree: coding-hermes, creative-hermes, research-hermes. Each has independent context windows. @mention a profile to route context. Per-profile visibility toggles. Cost-aware profile selection.

🌐 Cross-Server Federation

Other people bring their agents into group chats. Different Hermes instances federate — not everyone needs the same server. Each agent runs on its owner's infrastructure, knows their preferences, calendar, and knowledge base — participates in shared context. Two reply modes: public (visible to all) or direct (private to owner).

📴 Auto-Responder

Set status: "busy for 3 hours." Agent tells people who message you, pings you for important things, handles routine questions. Calendar-integrated: agent knows when your meetings are. Configurable interrupt thresholds.

Design Principles

  1. The agent's memory IS the tree. No invisible context window. What the agent sees is visible, navigable, and shareable.
  2. Files ARE the knowledge base. Don't re-upload. Files live on your Hermes instance. The chat app is the file browser.
  3. Cards are apps, Canopy is the OS. Each card has its own database, UI, state, and REST API. Skills rendered as interactive UI.
  4. The agent designs its own interface. The agent surfaces what's relevant — calendar, search results, code output — as cards. You customize per thread.
  5. Offline-first, not offline-afterthought. Local-first with CRDTs. Delta sync. Works on planes, spotty connections, anywhere.
  6. Seven deployment modes, one experience. Native app, web PWA, tunnel, P2P, message queue, self-hosted relay, SaaS relay. Same UI, same data.
  7. Git-native data. Card data is JSONL files. Diffable. Version-controlled. No binary blob databases.

Non-Goals (Explicit)

  • ❌ Not replacing Obsidian, Hermes, or your existing chat apps — it's the chat app FOR that stuff
  • ❌ Not Electron — WebUI uses system WebView, tiny binary
  • ❌ Not WebSockets primary — SSE for resilience
  • ❌ Not raw SQL from cards — pgREST fixed API only
  • ❌ Not SQLite for cards — DuckDB + JSONL
  • ❌ Not replacing human-to-human chat — this is for human-agent collaboration
