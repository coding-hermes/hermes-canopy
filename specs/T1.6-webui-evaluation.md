# T1.6 — WebUI Native App Evaluation

> **Status:** Complete
> **Date:** 2026-07-20
> **Task:** Evaluate WebUI library (webui.me, github.com/webui-dev/webui) for native desktop packaging
> **Output:** Recommendation for native app packaging approach

---

## 1. Executive Summary

**Recommendation: Wails v3 (primary) + Go `embed` with localhost HTTP server (fallback)**

WebUI uses the system browser as a GUI — it opens a browser tab and controls it via a lightweight C library. This is clever for tools/utilities but wrong for Canopy. Canopy is a collaboration surface that needs to feel like a native application, not a browser tab. WebUI's browser-chrome UI (address bar, tabs, bookmarks) breaks the immersive agent-collaboration experience. Its WebView mode (added later) is underdeveloped compared to dedicated WebView frameworks.

Wails embeds the system WebView (WebView2 on Windows, WKWebView on macOS, WebKitGTK on Linux) — no browser chrome, true native window, 10-30MB bundle. It's Go-native, mature (28k+ stars, v3 in alpha), and actively maintained. It's the Go equivalent of Tauri.

For MVP (PWA + local server), Go's built-in `embed` package + `net/http` server + browser launch is zero-dependency and sufficient. Wails becomes the upgrade path for native packaging post-MVP.

## 2. Candidate Evaluation

### 2.1 WebUI (webui-dev/webui)

| Dimension | Assessment |
|-----------|-----------|
| **GitHub Stars** | ~4.1k |
| **License** | MIT |
| **Language** | C core + Go bindings (webui-dev/go-webui) |
| **Approach** | Uses any installed web browser as GUI (or WebView mode) |
| **Binary Size** | Tiny — ~200KB static library, no embedded runtime |
| **Startup Time** | Fast — opens browser tab, negligible library overhead |
| **Memory** | Browser's memory (500MB+ for Chrome, less for others) |
| **Cross-Platform** | Windows, macOS, Linux |
| **WebView Mode** | Added in later releases — supports true WebView (no browser chrome) |
| **Go Bindings** | `github.com/webui-dev/go-webui/v2` — CGo wrapper |
| **JS↔Go Bridge** | Two-way bindings: Go functions callable from JS, JS callable from Go |

**Strengths:**
- Extremely small binary footprint
- Works anywhere a browser exists
- Simple API — `window.Show("<html>")` or `window.Show("http://localhost:8080")`
- Cross-platform without platform-specific SDKs
- Two-way JS↔Go communication

**Weaknesses:**
- **Browser mode (default):** Opens in user's default browser with full browser chrome — address bar, tabs, bookmarks. Not a native app experience. User can accidentally navigate away.
- **WebView mode:** Less mature — "Proposed by nater0000" per release notes, added as secondary feature. Missing key controls (devtools disable requested in issue #578, April 2025).
- **No window management:** No native menus, dialogs, tray icons, window positioning, or multi-window support.
- **Browser dependency:** Requires a modern browser installed. On headless Linux servers, this means installing Chromium.
- **No app packaging:** Produces a binary, not a .app/.exe/.deb installer. No code signing, no app store compliance.
- **Community size:** 4.1k stars vs Wails' 28k — smaller ecosystem, fewer examples.
- **CGo dependency:** C compilation toolchain required for builds.

**Verdict:** Good for developer tools and utilities where browser chrome is acceptable (CLI wrappers, config editors, monitoring dashboards). NOT suitable for a consumer-facing collaboration application that must feel native.

### 2.2 Wails (wailsapp/wails)

| Dimension | Assessment |
|-----------|-----------|
| **GitHub Stars** | ~28k |
| **License** | MIT |
| **Language** | Go-native (no CGo for core) |
| **Approach** | Embeds system WebView — true native window |
| **Bundle Size** | 10-30MB (Go binary + frontend assets) |
| **Startup Time** | Sub-500ms (per Wails benchmarks) |
| **Memory** | ~10MB idle + WebView (significantly less than Electron) |
| **Cross-Platform** | Windows (WebView2), macOS (WKWebView), Linux (WebKitGTK) |
| **Go Bindings** | Native — Wails IS Go |
| **JS↔Go Bridge** | Built-in IPC — Go functions exposed to frontend, events in both directions |
| **Version** | v2 stable, v3 in alpha (adds multi-window, new menu API, better IPC) |

**Strengths:**
- **True native feel:** System window with no browser chrome. Appears in dock/taskbar as a native app.
- **Go-native:** No CGo, no foreign toolchain. Builds with standard Go tooling.
- **Built-in features:** Native menus, dialogs, system tray, window management, drag-and-drop, file associations.
- **Frontend Agnostic:** React, Vue, Svelte, vanilla JS — anything that produces static files works.
- **Active development:** v3 alpha adds multi-window (critical for Canopy's side panels, approval panel, context manifest).
- **Production users:** Listed on wails.io — real companies shipping desktop apps.
- **Packaging:** `wails build` produces platform-native binaries. Community tooling for installers.
- **Documentation:** Comprehensive docs at wails.io, migration guides v2→v3.

**Weaknesses:**
- **WebKitGTK dependency on Linux:** Requires `libwebkit2gtk-4.1-dev` to be installed. This is standard on desktop Linux but may be missing on servers or minimal installs.
- **v3 not yet stable:** Multi-window and other Canopy-critical features are in alpha.
- **Larger than WebUI:** 10-30MB vs WebUI's 200KB — but still 10x smaller than Electron.
- **No mobile:** Desktop-only. PWA covers mobile. Not a conflict with Canopy's architecture.

**Verdict:** The correct choice for Canopy's native desktop packaging. Go-native, mature ecosystem, true native windows, active development. The v3 multi-window support directly maps to Canopy's UI architecture (main tree view, approval side panel, context manifest panel, search sidebar).

### 2.3 Tauri (tauri-apps/tauri)

| Dimension | Assessment |
|-----------|-----------|
| **GitHub Stars** | ~95k |
| **License** | MIT/Apache 2.0 |
| **Language** | Rust backend |
| **Approach** | Embeds system WebView |
| **Bundle Size** | Sub-600KB core + Rust binary |
| **Startup Time** | Fast |
| **Cross-Platform** | Same WebView targets as Wails |

**Verdict:** Excellent framework, but wrong language. Canopy's backend is Go. Adding Rust to the stack for desktop packaging adds a foreign toolchain, build complexity, and a second language to maintain. Wails achieves the same result in Go.

### 2.4 NeutralinoJS

| Dimension | Assessment |
|-----------|-----------|
| **GitHub Stars** | ~7.5k |
| **Language** | Language-agnostic (WebSocket-based IPC) |
| **Approach** | Embeds system WebView |

**Verdict:** Lighter than Wails but Go integration requires running a separate Go server and communicating via WebSocket. Wails' native Go IPC is cleaner. Neutralino's smaller community and slower development pace add risk.

### 2.5 Go `embed` + Localhost Server (Zero-Dependency)

| Dimension | Assessment |
|-----------|-----------|
| **Approach** | `//go:embed` frontend build into binary, `net/http` serves it, `browser.Open()` launches browser |
| **Binary Size** | Frontend size + Go binary (8-15MB typical) |
| **Startup Time** | ~200ms |
| **Memory** | ~15MB + browser |
| **Cross-Platform** | Yes — Go's stdlib is cross-platform |
| **Dependencies** | Zero external dependencies |

**Strengths:**
- Zero external dependencies — Go stdlib only.
- Single binary with embedded frontend.
- No CGo, no WebView SDKs, no platform-specific linking.
- Works everywhere Go compiles.
- Trivially Docker-packaged.
- Fastest to implement — `go:embed` + `http.FileServer` + `browser.OpenURL`.

**Weaknesses:**
- Opens in browser, not native window (same browser-chrome issue as WebUI browser mode).
- No IPC bridge — frontend communicates via HTTP to the local server.
- No window management, menus, tray icons.
- User can close the browser tab and lose the application context.
- No app store deployment path.

**Verdict:** Perfect for MVP. Canopy is already a PWA — the Go binary serves the PWA and provides the SSE/REST API. The user opens it in their browser. This is zero additional work beyond what's already spec'd. Wails is the post-MVP upgrade path when native packaging becomes a requirement.

| Feature | WebUI | Wails v3 | Go embed | Tauri | Neutralino |

aGo embed + localhost is not a "native packaging" solution — it's the PWA approach Canopy already plans. Included as the zero-effort baseline.

## 4. WebView Compatibility Matrix

All WebView-based frameworks (Wails, Tauri, WebUI-WebView-mode) share the same underlying platform WebViews:

| Platform | WebView Engine | Support Level | Notes |
|----------|---------------|---------------|-------|
| **Windows 10+** | Edge WebView2 | Excellent | Pre-installed on Win 11, evergreen auto-update on Win 10 |
| **Windows 7/8** | Edge WebView2 (manual install) | Good | Requires user to install WebView2 runtime |
| **macOS 10.13+** | WKWebView | Excellent | Built into macOS, consistent rendering |
| **Linux (GTK)** | WebKitGTK | Good | Requires `libwebkit2gtk-4.1-dev`, standard on desktop distros |
| **Linux (Qt)** | QtWebEngine | Limited | Wails doesn't support; Tauri community effort |
| **Headless Linux** | None | N/A | Can't use WebView — fall back to browser mode or PWA |

**Canopy's Linux target:** Bane runs Ubuntu 26.04 LTS (GTK-based). WebKitGTK is available via apt. This is the primary development and deployment target.

## 5. Architecture Impact

### 5.1 Go Embed Approach (MVP Path)

```
┌─────────────────────────────────────────────┐
│ Single Binary: canopyd                       │
│                                              │
│  canopyd serve --port 8080                   │
│  ├── //go:embed web/dist/ → http.FileServer  │
│  ├── REST API handlers (/api/...)            │
│  └── SSE Hub (/events/...)                   │
│                                              │
│  User opens http://localhost:8080 in browser │
└─────────────────────────────────────────────┘
```

Zero code changes from the backend spec. The frontend build output (`web/dist/`) is embedded at compile time. `canopyd serve` serves both the API and the PWA from the same port. This is a single-binary deployment.

### 5.2 Wails Path (Post-MVP)

```
┌─────────────────────────────────────────────┐
│ Wails App: canopyd + Wails runtime           │
│                                              │
│  Go backend (canopyd logic)                  │
│  ├── Wails IPC bridge                        │
│  │   ├── Callable from frontend:             │
│  │   │   - CreateNode, DeleteNode, ...       │
│  │   │   - GetApprovalQueue, ...             │
│  │   │   - SSE → Wails events                │
│  │   └── Events to frontend:                 │
│  │       - NodeAdded, NodeUpdated, ...       │
│  │       - ApprovalChanged, UserJoined       │
│  ├── REST API (still available for PWA)      │
│  └── SSE Hub (still available for PWA)       │
│                                              │
│  Native window (no browser chrome)           │
│  ├── Main tree view                          │
│  ├── Approval side panel (wails multi-window)│
│  ├── Context manifest panel                  │
│  └── Search sidebar                          │
└─────────────────────────────────────────────┘
```

The transition from Go embed to Wails is additive, not a rewrite. The REST API and SSE Hub remain unchanged — they serve the PWA fallback. Wails adds a native IPC bridge and native window management on top.

### 5.3 What Changes Between MVP and Wails

| Component | MVP (Go embed) | Wails |
|-----------|---------------|-------|
| Backend logic | Unchanged | Unchanged |
| REST API | Direct HTTP calls | HTTP calls + Wails IPC (dual path) |
| SSE events | EventSource in browser | Wails events (native IPC) |
| Frontend | React PWA | Same React code, Wails runtime instead of browser |
| Window chrome | Browser tab | Native OS window |
| Multi-window | No (browser tabs) | Yes (Wails v3 multi-window) |
| App packaging | None | `wails build` → .exe/.app/.deb |
| Build toolchain | Go + Node | Go + Node + Wails CLI |
| Deployment | Single binary | Single binary + WebView runtime check |

## 6. Decision

### Phase 1 (MVP): Go `embed` + Localhost Server

- Zero additional dependencies
- Single binary: `canopyd serve` serves everything
- PWA in browser — Canopy is already designed as a PWA
- Ships immediately with zero native-packaging investment
- User experience: browser tab — acceptable for MVP technical users

### Phase 2 (Post-MVP Native): Wails v3

- Upgrade when Wails v3 is stable (currently alpha)
- True native window with system chrome
- Multi-window support for Canopy's panel architecture
- Native menus, tray icon, app packaging
- Backend code reuses 100% — only frontend wiring changes
- Upgrade path: add `wails` Go build tags, dual-compile (embed mode + Wails mode)

### Not Recommended: WebUI

WebUI's design philosophy (use browser as GUI) conflicts with Canopy's need for a native collaboration surface. Its WebView mode is underdeveloped and lacks multi-window support. The 4.1k-star community is an order of magnitude smaller than Wails'. The browser-chrome experience is acceptable for developer tools but wrong for a consumer-facing application where context immersion matters.
