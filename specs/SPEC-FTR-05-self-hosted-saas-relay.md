## 1. Purpose

Define the exact implementation contract for Canopy's relay service layer — the component that enables **self-hosted** (home server behind NAT) and **SaaS** (managed cloud) deployment modes. The relay service is not a single adapter implementation but a **coordination layer** that includes registration, discovery, authentication, session management, multi-tenant isolation, deployment-specific configuration, and the self-hosted relay binary protocol specified in T1.8 §3.5.

A Go worker reading this document can implement:
- `RelayService` — the top-level service coordinating all relay operations
- `RelayHub` — the SaaS-side relay server managing 10,000+ concurrent sessions
- `SelfHostedRelay` — the self-hosted relay server (TCP/QUIC binary protocol from T1.8 §3.5)
- `RelayRegistry` — tenant/instance registration and discovery
- `DeploymentConfigManager` — configurable deployment-mode parameters
- All HTTP handlers, SSE event wiring, DDL tables, and the `--relay-listen` / `--relay-connect` CLI flags

without making transport-design decisions. The relay layer is one of five `TransportAdapter` implementations defined in SPEC-FTR-04, but it is the **only transport that is also a self-hosted service** — it combines a transport adapter (client side) with a relay server (server side) plus SaaS multi-tenant infrastructure.

**Key insight:** The self-hosted relay and SaaS relay share the same binary wire protocol (T1.8 §3.5) but differ in registration, authentication, discovery, and scaling characteristics. This spec defines the shared core and the deployment-specific extensions.

## 2. Design Decisions

| # | Decision | Choice | Rationale |
|---|----------|--------|-----------|
| 1 | Relay service vs embedded adapter | Separate `RelayService` struct that manages `RelayAdapter` (client) and `RelayHub` (server) plus `RelayRegistry` (registration) as sub-components | The relay adapter in SPEC-FTR-04 manages a single connection. The relay service manages lifetimes, registrations, and multi-tenant concerns. A single struct with composed sub-components avoids package sprawl while keeping the adapter interface clean. |
| 2 | Relay service location | `package relay` in `internal/relay/` — imports `package transport` for `TransportAdapter`, `Message`, `ConnectOptions` | The relay service is not inside `package transport` because it has concerns (registration, tenant management) that no other transport has. All other transports (SSE, WebRTC, NATS, Redis) are pure adapter implementations. |
| 3 | Self-hosted registration | Out-of-band: user configures `canopyd --relay-listen tcp://0.0.0.0:9443` and optionally registers with a SaaS discovery URL. No automatic public registration. | Self-hosted instances are inherently private. The admin chooses whether to make the relay discoverable. Out-of-band registration respects that choice. |
| 4 | SaaS registration | API-based: `POST /api/v1/instances/register` with a provisioning token generated by the SaaS admin console | Programmatic registration enables automated deployments (Docker, Ansible, Terraform). The provisioning token is a one-time-use JWT scoped to the tenant account. |
| 5 | Tenant isolation | Per-tenant `Instance` with dedicated database schema (PostgreSQL schema per tenant), isolated `Message` routing, and separate rate-limit buckets | Schema-per-tenant prevents cross-tenant data leakage at the database level. Separate routing prevents a misconfigured relay from delivering Tenant A messages to Tenant B. Separate rate-limit buckets prevent Tenant A's traffic burst from starving Tenant B. |
| 6 | Instance identity | UUIDv7 instance_id generated at first registration; Ed25519 keypair for HMAC signing of relay frames | UUIDv7 matches the existing Canopy identity model. Ed25519 keypair gives each instance a stable cryptographic identity for authenticating frames. The public key is stored in the relay registry; the private key stays on the instance. |
| 7 | Connection authentication | Two-layer: (1) TLS client certificate at transport layer, (2) HMAC-SHA256 of HELLO frame with instance's Ed25519-derived shared secret | TLS provides channel authentication. HMAC provides message-layer authentication that survives intermediary proxies. Dual-layer prevents replay attacks if TLS is terminated at a proxy. |
| 8 | Session limit | 10,000 concurrent sessions per SaaS relay node; 500 per self-hosted relay node | SaaS relay requires higher limits for multi-tenant scaling. Self-hosted relay runs on home server hardware (Raspberry Pi, NAS, homelab VM) where 500 sessions is the practical ceiling. Configurable via `--max-sessions` on both. |
| 9 | Heartbeat for relay | 30s PING/PONG over the relay connection. 90s timeout → connection dead → cleanup. | Matches the relay adapter heartbeat from T1.8 §3.5. 30s interval is frequent enough for timely disconnect detection without excessive bandwidth on constrained links. |
| 10 | SaaS discovery endpoint | `GET /api/v1/relays` — returns prioritized list of available relay nodes for a tenant, ordered by load and geographic proximity (future: latency measurement) | Enables SaaS load balancing without clients needing to know the relay topology. The list is tenant-scoped — each tenant sees only their assigned relay nodes. |
| 11 | Self-hosted discovery | mDNS (`_canopy-relay._tcp.local`) on LAN. No internet-scale auto-discovery. | Self-hosted instances on the same LAN can discover each other without configuration. For cross-LAN federation, use SPEC-FTR-02's manual invite-link pattern. |
| 12 | Deployment mode auto-detection | Same `TransportSelector` probe from SPEC-FTR-04 §2 Decision 19: loopback check → mDNS scan → STUN query → public IP match → air-gapped fallback | Reuses the existing probe. The relay service is initialized only when the detected mode requires it (self-hosted: relay enabled; SaaS: relay always available; local/air-gapped: relay optional). |
| 13 | Relay configuration persistence | `relay_config` table in PostgreSQL: per-instance of `{instance_id, listen_addr, max_sessions, heartbeat_secs, hmac_key_rotated_at, deployment_mode, enabled}` | Configuration is durable across restarts. The relay service reads this table at startup and applies the stored configuration. Changes via `PUT /api/v1/relays/config` update the table and hot-reload the affected sub-components. |
| 14 | HMAC key rotation | Automatic: `RelayService` rotates the HMAC key every 7 days. Old key retained for 48h grace window for inflight frames. Rotation triggered by time, not traffic volume. | Regular rotation limits the blast radius of a key compromise. The 48h grace window ensures that frames signed with the old key during a network partition are still verifiable when delivered. Rotation is logged to SPEC-DM-03 audit trail. |
| 15 | SaaS billing integration | Billing is out of scope for this spec. The `Instance` table includes a `tier` field (enum: `free`, `pro`, `enterprise`) that maps to resource limits. Billing logic is deferred to DEPLOY-04. | The relay layer enforces resource limits per tier (max sessions, max messages/sec, max storage for offline queue). The billing system that sets those limits is a separate concern. |
| 16 | Offline message queue for relay | Per-peer ring buffer (10,000 messages) in memory, backed by PostgreSQL for persistence. Same mechanism as SPEC-FTR-04 §4 Node 1. | Reuses the existing `MessageQueue` from SPEC-FTR-04's `ConnectionManager`. The relay adapter's offline queue is identical to every other transport's queue — no special-casing needed. |
| 17 | Relay node health | `Health()` pings the relay listener (TCP connect or QUIC ping). Failure → transport degraded per SPEC-FTR-04 §5. SaaS relay additionally publishes health metrics to a `/health` HTTP endpoint for load balancers. | Standard health-check pattern. The SaaS `/health` endpoint is consumed by the discovery service to exclude unhealthy nodes from `GET /api/v1/relays` responses. |
| 18 | Connection draining on shutdown | `RelayService.Shutdown(ctx)` → stop accepting new connections → signal existing connections with BYE frame → wait up to 30s for inflight message flush → close all connections | Graceful shutdown prevents message loss when `canopyd` restarts for upgrades or configuration changes. The 30s drain window is configurable via `--drain-timeout`. |
| 19 | Rate limiting per tenant | Separate token-bucket rate limiter per tenant: 1000 messages/sec inbound, 500 outbound. Burst: 2× for 5 seconds. | Same token-bucket design as SPEC-FTR-04 §2 Decision 18, but scoped per tenant instead of per-connection. Prevents one tenant's traffic from affecting another's latency. |
| 20 | Relay protocol versioning | Version field in `HELLO` frame (uint16). Backward compatible: server accepts any version ≥ client version. Version negotiation fails gracefully with `ErrTransportMismatch` and a `min_supported_version` hint. | Ensures smooth upgrades across a fleet of self-hosted and SaaS relay nodes running different canopyd versions. The version field is the `features bitmap` from T1.8 §3.5 augmented with major.minor. |

## 3. Go Interface Definitions

The following package is syntactically compilable Go. All types are defined in `package relay`. The `TransportAdapter` for relay lives in `package transport` (SPEC-FTR-04); this package depends on `package transport` but `package transport` never imports `package relay`.

### 3.1 Relay Service Interface and Core Types

```go
package relay

import (
    "context"
    "crypto/ed25519"
    "net"
    "time"

    "github.com/google/uuid"
    "canopyd/internal/transport"  // SPEC-FTR-04 TransportAdapter, Message, ConnectOptions
)

// RelayService is the top-level service managing all relay operations.
// It composes RelayHub (server), RelayAdapter (client via transport package),
// and RelayRegistry (registration/discovery).
type RelayService struct {
    hub      *RelayHub
    registry *RelayRegistry
    config   *DeploymentConfig

    // sessionLimit is the maximum concurrent sessions for this relay node.
    sessionLimit int

    // drainTimeout is the maximum time to wait for inflight message flush on shutdown.
    drainTimeout time.Duration

    // hmacKey is the current HMAC-SHA256 key for frame authentication.
    // Rotated automatically every 7 days via rotateHMACKey().
    hmacKey     []byte
    hmacKeyID   uint32
    hmacKeyPrev []byte      // previous key, retained for 48h grace window
    hmacKeyPrevSet time.Time

    // startedAt records when the relay service was initialized.
    // Used by Health() to report uptime.
    startedAt time.Time

    mu sync.RWMutex
}

// NewRelayService creates a relay service from deployment config.
// Loads config from relay_config table, initializes sub-components.
// Returns nil if relay is not configured (deployment mode doesn't need it).
func NewRelayService(ctx context.Context, cfg *DeploymentConfig) (*RelayService, error)

// Start begins accepting connections (server mode) or connecting to a
// configured relay address (client mode). In SaaS mode, also registers
// with the discovery endpoint.
func (rs *RelayService) Start(ctx context.Context) error

// Shutdown gracefully stops the relay service: stops accepting new connections,
// signals BYE to active sessions, drains message queues, closes all connections.
func (rs *RelayService) Shutdown(ctx context.Context) error

// Health returns the aggregate health of all relay sub-components.
func (rs *RelayService) Health(ctx context.Context) *RelayHealth

// RegisterTenant provisions a new tenant instance. Returns the instance's
// registration token. Only available in SaaS mode.
func (rs *RelayService) RegisterTenant(ctx context.Context, req *RegisterRequest) (*RegisterResponse, error)

// GetRelayNodes returns the prioritized list of relay nodes for a tenant.
// Only available in SaaS mode.
func (rs *RelayService) GetRelayNodes(ctx context.Context, tenantID uuid.UUID) ([]RelayNodeInfo, error)

// DeploymentConfig represents the relay service configuration for a deployment.
type DeploymentConfig struct {
    // Mode is the detected deployment mode.
    // Only relay-relevant modes: SelfHosted, SaaS, AirGapped.
    Mode transport.DeploymentMode

    // ListenAddr is the address to listen on for incoming relay connections.
    // In server mode: "tcp://0.0.0.0:9443" or "quic://0.0.0.0:9443"
    // In client mode: ""
    ListenAddr string

    // ConnectAddr is the relay address to connect to (client mode).
    // Only set when this instance is a relay client connecting to a relay server.
    ConnectAddr string

    // MaxSessions is the maximum concurrent relay sessions.
    // Default: 500 (self-hosted), 10000 (SaaS).
    MaxSessions int

    // HeartbeatSecs is the interval between PING frames.
    // Default: 30s.
    HeartbeatSecs int

    // DrainTimeout is the maximum time to wait for graceful shutdown.
    // Default: 30s.
    DrainTimeout time.Duration

    // TLS enables TLS 1.3 for the relay listener.
    // Self-hosted: optional. SaaS: required.
    TLS *TLSConfig

    // HMACKeyRotateInterval is how often to rotate the HMAC key.
    // Default: 7 days.
    HMACKeyRotateInterval time.Duration

    // TenantID is set in SaaS mode.
    TenantID uuid.UUID

    // InstanceID uniquely identifies this relay node.
    InstanceID uuid.UUID

    // PrivateKey is the Ed25519 identity key for this instance.
    // Generated at first registration; persisted in instance_identity file.
    PrivateKey ed25519.PrivateKey

    // DiscoveryURL is the SaaS discovery endpoint URL.
    // Set only in SaaS mode: "https://canopy.example.com/api/v1/relays"
    DiscoveryURL string
}

// TLSConfig configures TLS for relay connections.
type TLSConfig struct {
    CertFile string
    KeyFile  string
    CAFile   string        // for client certificate verification
    Mutual   bool          // require client certificates
}
```

### 3.2 RelayHub — SaaS and Self-Hosted Relay Server

```go
// RelayHub manages incoming relay connections on the relay server side.
// In SaaS mode, it multiplexes across tenants. In self-hosted mode,
// it serves a single tenant.
type RelayHub struct {
    listener net.Listener         // TCP or QUIC listener
    sessions map[string]*RelaySession  // session_id -> session
    sessionMu sync.RWMutex

    hmacKey     []byte
    hmacKeyID   uint32
    hmacKeyPrev []byte

    // sessionCounter tracks total sessions for limit enforcement.
    sessionCounter int32

    // sessionLimit is the max concurrent sessions.
    sessionLimit int32

    // tenantRateLimiters maps tenant_id -> token bucket.
    tenantRateLimiters map[string]*rateLimiter
}

// RelaySession represents a single relay connection.
type RelaySession struct {
    ID          string          // unique session ID (UUIDv7)
    InstanceID  uuid.UUID       // connecting instance
    TenantID    uuid.UUID       // tenant (SaaS mode only)
    Conn        net.Conn        // underlying TCP/QUIC connection
    Established time.Time
    LastActivity time.Time

    // rxSeq / txSeq for sequence tracking per T1.8 §4.3
    rxSeq uint64
    txSeq uint64

    // messageQueue for offline buffering
    messageQueue *transport.MessageQueue

    // done channel signals session cleanup
    done chan struct{}
}

// Accept blocks until a new relay connection arrives, performs the HELLO
// handshake (T1.8 §3.5 Connection Flow), and registers the session.
func (hub *RelayHub) Accept(ctx context.Context) (*RelaySession, error)

// HandleConnection processes a fully established relay connection.
// Reads frames from conn, decodes them into Message structs, and
// dispatches to the sync engine via the registered callback.
func (hub *RelayHub) HandleConnection(ctx context.Context, session *RelaySession) error

// CloseSession terminates a relay session gracefully: sends BYE frame,
// drains message queue, removes from sessions map.
func (hub *RelayHub) CloseSession(sessionID string) error

// BroadcastToTenant sends a Message to all sessions belonging to a tenant.
// Used in SaaS mode for tree events that affect multiple instances of the same tenant.
func (hub *RelayHub) BroadcastToTenant(tenantID uuid.UUID, msg *transport.Message) error

// rateLimiter implements a token-bucket rate limiter per SPEC-FTR-04 §2 Decision 18.
type rateLimiter struct {
    tokens    float64
    maxTokens float64
    rate      float64    // tokens per second
    lastTick  time.Time
    mu        sync.Mutex
}

func (rl *rateLimiter) Allow() bool
func (rl *rateLimiter) AllowN(n int) bool
```

### 3.3 RelayRegistry — Instance Registration and Discovery

```go
// RelayRegistry manages instance registration, tenant assignment,
// and discovery of available relay nodes.
type RelayRegistry struct {
    db     *sql.DB          // PostgreSQL connection
    mode   transport.DeploymentMode
}

// Instance represents a registered relay instance (SaaS) or
// a self-hosted relay node (self-hosted mode).
type Instance struct {
    InstanceID  uuid.UUID         `json:"instance_id"`
    TenantID    uuid.UUID         `json:"tenant_id,omitempty"` // set in SaaS mode
    PublicKey   []byte            `json:"public_key"`          // Ed25519 public key
    ListenAddr  string            `json:"listen_addr"`
    Tier        string            `json:"tier"`               // "free", "pro", "enterprise" (SaaS)
    Enabled     bool              `json:"enabled"`
    ConnectedAt *time.Time        `json:"connected_at,omitempty"`
    CreatedAt   time.Time         `json:"created_at"`
    UpdatedAt   time.Time         `json:"updated_at"`
}

// RegisterRequest is the payload for registering a new relay instance.
type RegisterRequest struct {
    TenantID    uuid.UUID `json:"tenant_id"`     // SaaS: assigned by admin
    PublicKey   []byte    `json:"public_key"`    // Ed25519 public key
    ListenAddr  string    `json:"listen_addr"`   // public address of this relay node
    Tier        string    `json:"tier"`          // SaaS: "free", "pro", "enterprise"
    ProvisioningToken string `json:"provisioning_token"` // SaaS: one-time JWT
}

// RegisterResponse is returned after successful registration.
type RegisterResponse struct {
    InstanceID  uuid.UUID `json:"instance_id"`
    RelaySecret string    `json:"relay_secret"`  // pre-shared HMAC base key
    CreatedAt   time.Time `json:"created_at"`
}

// RelayNodeInfo is returned by the discovery endpoint.
type RelayNodeInfo struct {
    InstanceID  uuid.UUID `json:"instance_id"`
    ListenAddr  string    `json:"listen_addr"`
    Load        float64   `json:"load"`          // 0.0 (idle) to 1.0 (full)
    Region      string    `json:"region,omitempty"`
}

// RegisterInstance registers a relay instance.
// In SaaS mode, validates the provisioning token and assigns the tenant.
// In self-hosted mode, generates a new instance identity.
func (reg *RelayRegistry) RegisterInstance(ctx context.Context, req *RegisterRequest) (*RegisterResponse, error)

// GetAvailableRelays returns relay nodes for a tenant, ordered by load.
// Excludes disabled and unhealthy nodes. Only used in SaaS mode.
func (reg *RelayRegistry) GetAvailableRelays(ctx context.Context, tenantID uuid.UUID) ([]RelayNodeInfo, error)

// GetInstanceByID retrieves a single instance record.
func (reg *RelayRegistry) GetInstanceByID(ctx context.Context, instanceID uuid.UUID) (*Instance, error)

// UpdateInstanceHeartbeat updates the last-seen time for an instance.
// Called on every successful HELLO handshake.
func (reg *RelayRegistry) UpdateInstanceHeartbeat(ctx context.Context, instanceID uuid.UUID) error
```

### 3.4 DeploymentConfigManager

```go
// DeploymentConfigManager reads, writes, and hot-reloads relay configuration.
type DeploymentConfigManager struct {
    db *sql.DB
}

// LoadConfig reads the relay configuration from the database.
// Returns the default config if no row exists (relay disabled).
func (mgr *DeploymentConfigManager) LoadConfig(ctx context.Context) (*DeploymentConfig, error)

// SaveConfig writes relay configuration to the database.
// The relay service picks up changes via a reload callback.
func (mgr *DeploymentConfigManager) SaveConfig(ctx context.Context, cfg *DeploymentConfig) error

// ReloadConfig hot-reloads relay configuration without restarting.
// Updates HMAC key rotation parameters, session limits, and heartbeat interval.
func (mgr *DeploymentConfigManager) ReloadConfig(ctx context.Context, rs *RelayService) error

// DefaultSelfHostedConfig returns sensible defaults for a self-hosted relay:
//   MaxSessions: 500
//   HeartbeatSecs: 30
//   DrainTimeout: 30s
//   HMACKeyRotateInterval: 168h (7 days)
func DefaultSelfHostedConfig() *DeploymentConfig

// DefaultSaaSConfig returns sensible defaults for a SaaS relay node:
//   MaxSessions: 10000
//   HeartbeatSecs: 15
//   DrainTimeout: 60s
//   HMACKeyRotateInterval: 168h (7 days)
//   TLS: required
func DefaultSaaSConfig() *DeploymentConfig
```

## 4. DDL

### 4.1 `relay_config` — Per-Instance Relay Configuration

```sql
CREATE TABLE relay_config (
    instance_id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    listen_addr           TEXT NOT NULL DEFAULT '',          -- empty = client mode only
    connect_addr          TEXT NOT NULL DEFAULT '',          -- empty = server mode only
    max_sessions          INTEGER NOT NULL DEFAULT 500,
    heartbeat_secs        INTEGER NOT NULL DEFAULT 30,
    drain_timeout_secs    INTEGER NOT NULL DEFAULT 30,
    deployment_mode       TEXT NOT NULL DEFAULT 'self_hosted'  -- 'self_hosted', 'saas', 'air_gapped'
                           CHECK (deployment_mode IN ('self_hosted', 'saas', 'air_gapped')),
    tls_enabled           BOOLEAN NOT NULL DEFAULT false,
    tls_cert_file         TEXT,
    tls_key_file          TEXT,
    tls_ca_file           TEXT,
    tls_mutual            BOOLEAN NOT NULL DEFAULT false,
    hmac_key_rotated_at   TIMESTAMPTZ,
    hmac_key_id           INTEGER NOT NULL DEFAULT 0,
    enabled               BOOLEAN NOT NULL DEFAULT false,
    created_at            TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at            TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### 4.2 `relay_instances` — Registered Relay Instances (SaaS)

```sql
CREATE TABLE relay_instances (
    instance_id           UUID PRIMARY KEY,
    tenant_id             UUID NOT NULL REFERENCES tenants(tenant_id),
    public_key            BYTEA NOT NULL,                 -- Ed25519 public key (32 bytes)
    listen_addr           TEXT NOT NULL,
    tier                  TEXT NOT NULL DEFAULT 'free'
                           CHECK (tier IN ('free', 'pro', 'enterprise')),
    enabled               BOOLEAN NOT NULL DEFAULT true,
    load_factor           REAL NOT NULL DEFAULT 0.0,      -- 0.0 (idle) to 1.0 (full)
    region                TEXT,
    connected_at          TIMESTAMPTZ,
    last_heartbeat_at     TIMESTAMPTZ,
    created_at            TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at            TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_relay_instances_tenant ON relay_instances(tenant_id);
CREATE INDEX idx_relay_instances_heartbeat ON relay_instances(last_heartbeat_at);
```

### 4.3 `relay_sessions` — Active Relay Sessions

```sql
CREATE TABLE relay_sessions (
    session_id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    instance_id           UUID NOT NULL REFERENCES relay_instances(instance_id),
    tenant_id             UUID NOT NULL REFERENCES tenants(tenant_id),
    remote_addr           TEXT NOT NULL,
    established_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    last_activity_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    rx_messages           BIGINT NOT NULL DEFAULT 0,
    tx_messages           BIGINT NOT NULL DEFAULT 0,
    rx_bytes              BIGINT NOT NULL DEFAULT 0,
    tx_bytes              BIGINT NOT NULL DEFAULT 0,
    closed_at             TIMESTAMPTZ,
    close_reason          TEXT,                           -- 'graceful', 'timeout', 'error', 'admin'
    closed_by             TEXT                            -- 'system', 'admin', 'peer'
);

CREATE INDEX idx_relay_sessions_instance ON relay_sessions(instance_id);
CREATE INDEX idx_relay_sessions_active ON relay_sessions(closed_at) WHERE closed_at IS NULL;
```

### 4.4 `instance_identity` — Local Instance Identity File (not a DB table)

Persisted to `~/.canopy/instance_identity.json` (self-hosted) or provisioned at SaaS registration time:

```json
{
  "instance_id": "0191a8b2-7fff-7000-9000-000000000001",
  "private_key": "base64-encoded-ed25519-private-key",
  "public_key": "base64-encoded-ed25519-public-key",
  "created_at": "2026-07-22T12:00:00Z"
}
```

This file is created at first start if it does not exist. The private key is never transmitted over the network; only the public key is registered with the relay registry.

### 5.3 Relay Binary Protocol Frame (from T1.8 §3.5)

```
┌───────────────────────────────────────────────────────────────────┐
│                        RELAY FRAME (Binary)                        │
├───────────────────────────────────────────────────────────────────┤
│ Bytes 0-3:    Magic Number 0x43 0x41 0x4E 0x59 ("CANY")           │
│ Byte 4:       Protocol Version (major << 4 | minor)               │
│ Bytes 5-6:    Frame Type (0x01 HELLO, 0x02 HELLO_ACK, etc.)      │
│ Bytes 7-14:   Session ID (UUIDv7, 16 bytes → truncated to 8)     │
│ Bytes 15-22:  Sequence Number (uint64, big-endian)                 │
│ Bytes 23-24:  CBOR Payload Length (uint16, big-endian)            │
│ Bytes 25-26:  HMAC Algorithm (0x01 = HMAC-SHA256)                 │
│ Byte 27:      Flags (bit 0: encrypted, bit 1: compressed)         │
│ Bytes 28-45:  Reserved (18 bytes, zero-padded)                    │
│ Bytes 46-45+M: CBOR-encoded Payload                               │
│ Final 32:     HMAC-SHA256 (over bytes 0 to 45+M)                  │
├───────────────────────────────────────────────────────────────────┤
│ Total header: 46 bytes | HMAC: 32 bytes | Max frame: 1MB          │
└───────────────────────────────────────────────────────────────────┘
```

## 6. SSE Events

Following the event-stream pattern defined in SPEC-API-01 and SPEC-FTR-04 §5, the relay layer emits three event types over the authenticated SSE stream.

### 6.1 relay_status

**Trigger:** Connection state transitions for relay sessions (connecting → active → degraded → closed). Also emitted on `RelayService.Start()` and `RelayService.Shutdown()`.

```json
{
  "event": "relay_status",
  "data": {
    "relay_mode": "self_hosted",
    "state": "active",
    "session_count": 12,
    "max_sessions": 500,
    "instance_id": "0191a8b2-7fff-7000-9000-000000000001",
    "listen_addr": "tcp://0.0.0.0:9443",
    "uptime_secs": 86400,
    "last_hmac_rotation": "2026-07-22T12:00:00Z",
    "degraded": false
  }
}
```

### 6.2 relay_tenant_event (SaaS only)

**Trigger:** Tenant-level events: instance registered, instance deregistered, tier changed, rate limit exceeded.

```json
{
  "event": "relay_tenant_event",
  "data": {
    "event_type": "instance_registered",
    "tenant_id": "0191a8b2-7fff-7000-9000-000000000042",
    "instance_id": "0191a8b2-7fff-7000-9000-000000000099",
    "tier": "pro",
    "relay_addr": "tcp://relay-us-east-1.canopy.example.com:9443"
  }
}
```

`event_type` values: `"instance_registered"`, `"instance_deregistered"`, `"tier_changed"`, `"rate_limit_warning"`, `"rate_limit_exceeded"`, `"hmac_rotation_started"`, `"hmac_rotation_completed"`.

### 6.3 relay_session_event

**Trigger:** Individual relay session lifecycle: connected, disconnected, error.

```json
{
  "event": "relay_session_event",
  "data": {
    "session_id": "0191a8b2-7fff-7000-9000-000000000200",
    "event_type": "connected",
    "remote_addr": "203.0.113.42:44820",
    "instance_id": "0191a8b2-7fff-7000-9000-000000000001",
    "tenant_id": "0191a8b2-7fff-7000-9000-000000000042",
    "protocol": "tcp",
    "established_at": "2026-07-22T12:00:00Z"
  }
}
```

## 7. CLI Flags

The `canopyd` binary gains three new flags for relay configuration:

```
--relay-mode string          Relay mode: "disabled", "self_hosted", "saas", "air_gapped"
                             (default: "disabled")
--relay-listen string        Address to listen on for relay connections
                             Example: "tcp://0.0.0.0:9443"
--relay-connect string       Address to connect to as a relay client
                             Example: "tcp://relay.example.com:9443"
--max-relay-sessions int     Maximum concurrent relay sessions
                             (default: 500 for self_hosted, 10000 for saas)
--relay-heartbeat int        Heartbeat interval in seconds (default: 30)
--relay-drain-timeout int    Graceful shutdown drain timeout in seconds (default: 30)
--relay-tls-cert string      TLS certificate file path
--relay-tls-key string       TLS key file path
--relay-tls-ca string        CA certificate file for mutual TLS
--relay-tls-mutual           Enable mutual TLS (client certificates required)
```

CLI flags override config file and database values. If `--relay-mode` is `"disabled"` (default), the relay service is not started.

## 9. Edge Cases

| # | Edge Case | Description | Mitigation |
|---|-----------|-------------|------------|
| 1 | Relay server behind double NAT | Self-hosted relay behind carrier-grade NAT (CGNAT) where port forwarding is impossible | RelayAdapter detects CGNAT via the loopback + STUN probe (SPEC-FTR-04 §2 Decision 19). Falls back to outbound-only relay connection: the self-hosted instance connects to a SaaS relay node as a client instead of listening. The binary protocol is symmetric — server and client mode use the same frames. |
| 2 | SaaS relay node overload | All SaaS relay nodes in a region reach 95%+ load simultaneously | Discovery API excludes overloaded nodes. New tenants are assigned to the least-loaded region. Existing sessions are not disrupted — `RelayHub.BroadcastToTenant` sends a `relay_tenant_event` with `event_type: "capacity_warning"` and the overloaded node stays operational for existing sessions. The SaaS admin console triggers auto-scaling. |
| 3 | HMAC key rotation during active session | The relay service rotates the HMAC key while a session is actively exchanging frames | The 48h grace window (Design Decision 14) handles this: the HMAC key ID in the frame header identifies which key was used. The receiver checks the current key first, then the grace key. Frames signed with the old key are accepted for 48h post-rotation. After 48h, old-key frames are rejected with a `ErrAuthFailed` — the sender must reconnect. |
| 4 | Relay connection over severely constrained link | A self-hosted relay on a satellite or cellular backhaul with <100KB/s bandwidth | Bandwidth adaptation (SPEC-FTR-04 §2 Decision 12) applies: relay frames are not exempt. The binary wire protocol (CBOR + HMAC header) adds ~78 bytes overhead per frame — negligible even on constrained links. Content trimming to IDs-only at <100KB/s further reduces payload. The 30s heartbeat interval can be increased via config (`--relay-heartbeat 120`). |
| 5 | Tenant migration between SaaS relay nodes | A SaaS tenant's session must move from Relay Node 1 (us-east-1) to Relay Node 2 (eu-west-1) without message loss | Migration is a two-phase process: (1) Relay Server 1 sends a `relay_tenant_event` with `event_type: "migration_started"` to all connected instances, including the target relay address. (2) Each instance reconnects to the new relay node. Inflight messages at Relay 1 are forwarded to Relay 2 via the NATS backbone before the session is closed. The `SeqWatermark` from SPEC-FTR-04 §4.3 ensures continuity: the new relay node starts from `last_ack_seq + 1`. |
| 6 | Instance identity file corruption | The `instance_identity.json` file is corrupted or deleted on a self-hosted instance | On startup, if the file is unreadable, `RelayService` generates a new Ed25519 keypair and creates a new instance_id. Previous relay connections using the old key will be rejected with `ErrAuthFailed`. The instance must re-establish trust via out-of-band verification. For SaaS instances, a new registration via `POST /api/v1/relays/instances` creates a fresh identity. |
| 7 | Simultaneous registration race | Two provisioning tokens for the same tenant are redeemed simultaneously | `INSERT INTO relay_instances` with `ON CONFLICT (instance_id) DO NOTHING`. The provisioning token is consumed on first successful registration; the second attempt receives 409 Conflict with the existing instance_id. The caller can retry with a new token or use the existing instance_id to re-register (which updates the public key but retains the instance_id). |
| 8 | Relay session limit undercut | A misconfigured relay node sets `max_sessions` to 0, denying all connections | Validation in `PUT /api/v1/relays/config` rejects `max_sessions < 1` with 422 Unprocessable Entity. On startup, `DeploymentConfigManager.LoadConfig` clamps the value to the deployment default if it's out of range. CLI flag `--max-relay-sessions` with value 0 is rejected at parse time. |
| 9 | SaaS discovery with no available nodes | All relay nodes for a tenant are disabled or unhealthy | `GET /api/v1/relays` returns HTTP 503 with body `{"error": "no_available_relays", "message": "All relay nodes for this tenant are currently unavailable."}`. The client's `TransportSelector` treats this as a transport failure and falls back to the next transport in the chain per SPEC-FTR-04 §5. The client retries discovery with exponential backoff. |
| 10 | Protocol version mismatch during HELLO | A canopyd v1.0 instance tries to connect to a canopyd v2.0 relay server with incompatible frame format | HELLO frame includes `version` field (Design Decision 20). Server compares against its `min_supported_version`. If client version is lower, server responds with HELLO_ACK carrying `ErrTransportMismatch` and a `min_version` hint. Client logs an actionable error: "Relay server v2.0 requires minimum protocol v2 — upgrade canopyd to v2.0 or later." |
| 11 | TLS certificate expiry for SaaS relay | A SaaS relay node's TLS certificate expires, rejecting all incoming connections | `RelayService` checks certificate expiry on `Health()`. If expiring within 7 days, emits a `relay_status` event with `degraded: true` and `degradation_reason: "tls_cert_expiring_soon"`. If expired, `RelayService` disables the listener and the discovery API excludes the node. The operator is alerted via monitoring. The node continues to serve existing sessions until the last one disconnects. |
| 12 | Self-hosted relay on battery-powered device | A Raspberry Pi acting as a relay node experiences a brownout or voltage dip | The relay service does not introduce any pull or polling — it is purely push-based (accept connections, forward frames). A brief brownout causes TCP connections to drop, which triggers reconnection on the client side (exponential backoff per T1.8 §3.5). On restoration, the relay service resumes accepting connections. No persistent state is at risk because relay sessions are ephemeral and message queues are backed by PostgreSQL on a separate volume (or the Pi's SD card with WAL). |

| ARCHITECTURE.md (§10 SaaS, §11 Self-Host Guide) | Defines the deployment architecture that the relay service implements. SaaS mode (ARCHITECTURE.md §10) requires multi-tenant isolation, tenant discovery, and load-balanced relay nodes. Self-hosted mode (§11) requires single-tenant relay with optional NAT traversal. The relay service formalizes these architecture requirements into implementable interfaces. |

The relay service is the implementation of ARCHITECTURE.md's "Relay server: canopyd binary" line. Self-hosted mode deploys the relay as part of the single binary; SaaS mode deploys dedicated relay nodes managed by the relay service. |
