# T1.7 — Security Protocol: MLS-Only Architecture

> Status: COMPLETE | Date: 2026-07-20 | Author: coding-hermes-foreman
> Decision: MLS-only. No Signal Protocol. No hybrid negotiation layer.

## Executive Summary

Canopy adopts MLS (Messaging Layer Security, RFC 9420) as its sole encryption protocol. Signal Protocol is explicitly rejected. This is not a "MLS-primary, Signal-fallback" architecture — MLS handles every case including 1:1 conversations (MLS groups of 2). Single dependency, single code path, halved attack surface.

---

## 1. Decision Rationale

### 1.1 Canopy Conversations Are Inherently Multi-Participant

In Canopy, "1:1" does not exist. Every conversation has at minimum:

| Participant | Always Present? | Role |
|---|---|---|
| Human user | Yes | Conversation owner |
| Hermes agent | Yes | Context compiler, model caller, tool executor |
| Canopy system | Yes | Audit logging, context management, sync engine |
| Additional Hermes profiles | Often | Specialized profiles (@coding-hermes, @research-agent) |
| Invited friends/users | Post-MVP | Multi-user collaboration |

A "private thread" between you and your agent is actually a 3+ member group (you, agent, system). Signal's Double Ratchet is designed for two endpoints. Forcing it onto 3+ participants creates protocol mismatch from day one.

### 1.2 Single Dependency, Halved Attack Surface

| Approach | Dependencies | Attack Surface | Audit Burden |
|---|---|---|---|
| MLS + Signal hybrid | mls-rs + libsignal + negotiation layer | 2 protocol surfaces + glue code | 3 codebases |
| **MLS-only (selected)** | **mls-rs (CGo)** | **1 protocol surface** | **1 codebase** |

libsignal is a C library with a large surface area. mls-rs is pure Rust with formal verification tooling (hacspec). Adding both means doubling the professional audit cost and maintaining two cryptographic codebases in perpetuity.

### 1.3 Industry Trajectory

- MLS is RFC 9420 — ratified IETF standard (2023)
- WhatsApp is adopting MLS for encrypted group messaging
- Matrix is migrating to MLS for room encryption
- IETF MLS working group continues active development of extensions
- Signal Protocol has no IETF standardization path

Canopy is being built for 2027-2030. Betting on MLS aligns with the industry's 5-year trajectory.

### 1.4 MLS Handles Groups of 2 Just Fine

A "1:1" Canopy conversation is simply an MLS group with 2 members. The overhead vs Signal:

| Metric | MLS (group of 2) | Signal (Double Ratchet) | Difference |
|---|---|---|---|
| Handshake messages | 2 (Welcome + Commit) | ~4 (X3DH + PreKey) | MLS simpler |
| Per-message overhead | ~100 bytes (framing) | ~90 bytes (ratchet header) | Negligible |
| Key rotation | Commit message | Automatic per-message | Signal more frequent |
| Async operation | Built-in (PendingProposals) | Requires pre-key server | MLS better for offline |

There is no protocol-level disadvantage to using MLS for 2-person groups.

## 2. Architecture

### 2.1 Group Hierarchy

```
Tree "project-alpha" MLS Group (root)
├── Topic "auth-flow" MLS Subgroup
│   ├── Member: human (owner)
│   ├── Member: hermes-agent (system)
│   └── Member: coding-hermes (profile)
├── Topic "database-schema" MLS Subgroup
│   ├── Member: human (owner)
│   ├── Member: hermes-agent (system)
│   └── Member: research-agent (profile)
└── Topic "private-thread" MLS Subgroup
    ├── Member: human (owner)
    └── Member: hermes-agent (system)
        (MLS group of 2 — works perfectly)
```

- **Per-tree MLS group:** All members of a tree share one root group. Tree-level config (name, visibility, member list) encrypted in application messages within this group.
- **Per-topic MLS subgroup:** Topics with restricted membership derive a subgroup. Topic content (messages, cards, #references) encrypted within the topic's group.
- **Group derivation:** Per-topic subgroups are independent MLS groups, not cryptographic derivatives of the tree group. This avoids the group-key compromise → all-topic exposure chain.

### 2.2 CGo Binding Architecture

```
┌──────────────────────────────────────┐
│              canopyd (Go)             │
│  ┌────────────────────────────────┐  │
│  │   internal/encryption/mls.go   │  │
│  │   - GroupManager                │  │
│  │   - CreateGroup(treeID)         │  │
│  │   - AddMember(groupID, userID)  │  │
│  │   - RemoveMember(...)           │  │
│  │   - Encrypt(groupID, plaintext) │  │
│  │   - Decrypt(groupID, ciphertext)│  │
│  └───────────┬────────────────────┘  │
│              │ CGo                    │
│  ┌───────────▼────────────────────┐  │
│  │   mls-rs (Rust, static lib)    │  │
│  │   - openmls crate               │  │
│  │   - CoreGroup, MlsGroup         │  │
│  │   - Ciphersuite: MLS_128_DHKEMX │  │
│  │   - Credential: BasicCredential │  │
│  └────────────────────────────────┘  │
└──────────────────────────────────────┘
```

**Go surface (internal/encryption/mls.go):**

```go
package encryption

import "context"

// GroupManager manages MLS groups for trees and topics.
type GroupManager struct {
    store GroupStore // PostgreSQL-backed key package storage
}

// CreateGroup creates a new MLS group for a tree or topic.
// Returns the group ID and the creator's Welcome message.
func (m *GroupManager) CreateGroup(ctx context.Context, groupID string, members []MemberKey) (*Group, error)

// AddMember adds a member to an existing group, triggering key rotation.
// Returns a Welcome message for the new member and a Commit for existing members.
func (m *GroupManager) AddMember(ctx context.Context, groupID, memberID string) error

// RemoveMember removes a member, triggering key rotation for forward secrecy.
func (m *GroupManager) RemoveMember(ctx context.Context, groupID, memberID string) error

// Encrypt encrypts application message for the group.
func (m *GroupManager) Encrypt(ctx context.Context, groupID string, plaintext []byte) (ciphertext []byte, err error)

// Decrypt decrypts an application message from the group.
func (m *GroupManager) Decrypt(ctx context.Context, groupID string, ciphertext []byte) (plaintext []byte, err error)

// ProcessMessage processes an incoming MLS protocol message (Proposal, Commit, Welcome).
// Returns any application messages extracted during processing.
func (m *GroupManager) ProcessMessage(ctx context.Context, groupID string, msg []byte) ([]ApplicationMessage, error)
```

### 2.3 Key Rotation

| Event | Action | MLS Operation |
|---|---|---|
| Member joins tree/topic | Rotate group key, send Welcome to new member | Add proposal → Commit |
| Member leaves | Rotate group key (forward secrecy) | Remove proposal → Commit |
| Member key compromised | Rotate member's key, all other keys unchanged | Update proposal → Commit |
| Periodic (7 days) | Rotate group key proactively | ReInit or empty Commit |
| Credential expires | Rotate affected member's credential | Update proposal → Commit |

### 2.4 Server-Side Agent as Group Member

The Hermes agent is an explicit, visible member of every MLS group:

- **Present in member list:** All human members can see "hermes-agent" in the group roster
- **Decrypts messages:** Agent reads all messages to build context windows, detect topics, and respond
- **Not a silent observer:** Agent's membership is transparent and auditable
- **Search over encrypted data:** Agent decrypts → indexes → re-encrypts. Search index is not encrypted at rest (it's a local SQLite FTS index on the server).

This means the agent is NOT a "man in the middle" or "backdoor" — it's a first-class group member with documented, user-visible membership.

## 3. What Signal Protocol Would Have Added (And Why It's Rejected)

| Signal Feature | MLS Equivalent | Why MLS Is Better for Canopy |
|---|---|---|
| Double Ratchet (per-message PFS) | MLS epoch-based PFS (per-commit) | Per-message PFS is overkill for tree sync — messages are batched in deltas |
| X3DH key agreement | MLS Welcome + KeyPackage | MLS Welcome is simpler, async-friendly, no pre-key server needed |
| Sender Keys (group efficiency) | MLS native group operations | Sender Keys are an add-on hack; MLS has group operations as first-class |
| Sealed Sender (metadata hiding) | Not in base MLS (MLS extensions) | Metadata privacy not required for MVP; can add via MLS PrivateMessage extension |
| Phone-number identity | MLS Credential (any identity) | Canopy uses profile tokens, not phone numbers — MLS credentials are flexible |

**Key rejection:** Signal's "1:1 is a special case" assumption. In Canopy, nothing is 1:1. The model assumes every channel has ≥3 participants. Building a dual-protocol stack (MLS for groups, Signal for 1:1) creates a protocol negotiation surface that itself becomes an attack vector. An attacker who can force a downgrade from MLS to Signal can then exploit Signal's 2-party assumption in a 3-party context.

## 4. Security Properties

| Property | Guarantee | Mechanism |
|---|---|---|
| **Confidentiality** | Messages readable only by group members | MLS AEAD encryption (AES-128-GCM or ChaCha20-Poly1305) |
| **Forward secrecy** | New members can't read old messages | Key rotation on member add (epoch transition) |
| **Post-compromise security** | Old members can't read new messages | Key rotation on member remove (epoch transition) |
| **Authentication** | Messages verifiable to sender | MLS FramedContentAuthData (signature over message + context) |
| **Group agreement** | All members agree on group state | MLS transcript hash chain |
| **Asynchronous operation** | Members offline during key ops | MLS PendingProposals queue (server holds until member comes online) |

## 5. Implementation Plan

### MVP (Local-Only, No Encryption)

Canopy MVP is single-user, local server + PWA. No network transport beyond localhost. Encryption is unnecessary — all data is on one machine.

### Post-MVP Encryption Addition (Phase 3d SPEC-FTR-03)

1. **mls-rs CGo binding:** Wrap the openmls Rust crate with a C-compatible FFI, expose via CGo in `internal/encryption/mls.go`
2. **KeyPackage storage:** PostgreSQL table for MLS key packages, indexed by user/profile ID
3. **Group creation:** On tree creation, create MLS group with owner + hermes-agent as members
4. **Topic subgroups:** On topic creation with restricted members, create independent MLS subgroup
5. **Message encryption:** All node content encrypted with topic's group key before writing to PostgreSQL
6. **Key rotation:** On member add/remove (post-MVP multi-user), trigger MLS commit + Welcome

### Dependencies

| Dependency | Version | Purpose |
|---|---|---|
| openmls (mls-rs) | 0.7+ | MLS protocol implementation in Rust |
| CGo | Go 1.22+ stdlib | Rust↔Go FFI bridge |
| PostgreSQL | 16+ | KeyPackage storage, group state persistence |

## 6. Threat Model & Attack Surface

### Server-Side Agent as Trusted Member

The agent is a group member — not an eavesdropper. This is a design choice, not a security flaw:

- The user explicitly grants the agent access to their conversations
- The agent's job requires reading messages (context assembly, topic detection)
- All group members see "hermes-agent" in the member roster
- Audit log records every decryption event

### Metadata Visibility

MLS encrypts message content but does not hide group metadata (member list, message timing, group structure) from the delivery service. In Canopy's self-hosted/local deployment:

| Deployment Mode | Metadata Risk | Mitigation |
|---|---|---|
| **Local** (MVP) | None — service runs on user's machine | N/A |
| **Self-hosted relay** | Relay sees group membership, timing | Relay is user-operated; trust boundary is the user |
| **SaaS relay** | Relay provider sees metadata | Post-MVP: MLS PrivateMessage extension for metadata encryption |

### Forward Secrecy Granularity

MLS provides epoch-based forward secrecy (after each Commit), not per-message PFS. For Canopy's tree sync pattern:

- Tree deltas are batched (multiple messages per SSE event)
- Per-message PFS would require a ratchet step per node — prohibitive for 10K+ node trees
- Epoch-based PFS (Commit on member change + periodic rotation) is the correct tradeoff

## 7. Alternatives Considered

### 7.1 MLS + Signal Hybrid (Rejected)

A negotiation layer that selects MLS for groups and Signal for 1:1:

| Pro | Con |
|---|---|
| Uses best protocol per use case | Protocol detection is an attack surface |
| Signal is more battle-tested for 1:1 | "1:1" doesn't exist in Canopy — everything is 3+ participants |
| Familiar to security auditors | Two dependencies, two audit targets, two update pipelines |
| | Negotiation layer doubles implementation complexity |

### 7.2 No Encryption Until Multi-User (Considered, Rejected as Strategy)

Delay all encryption decisions until multi-user deployment:

| Pro | Con |
|---|---|
| Simplifies MVP | Architecture may require redesign when encryption is added later |
| Reduces initial build time | Data model may assume plaintext (e.g., search indexes) |
| | "Add encryption later" is the #1 cause of insecure systems |

**Decision:** Design for encryption from day one (MLS group IDs in the data model, encrypted content columns), implement in MVP as "no-op encryption" (pass-through), activate real encryption post-MVP. No data model migration needed.

### 7.3 Custom Protocol (Rejected Immediately)

Building a custom encryption protocol:

- Violates "don't roll your own crypto" — the most fundamental rule in security engineering
- No external audit trail
- No formal verification
- No interop with other systems
