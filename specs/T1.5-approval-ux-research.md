# T1.5 — Approval UX Research

> Status: COMPLETE | Date: 2026-07-20 | Author: coding-hermes-foreman

## Executive Summary

**Recommendation: GitHub-style triage panel + Linear notification discipline + Google Docs per-item accept/reject granularity.**

Approval UX for Canopy is uniquely high-stakes: an agent is listening to all input and can execute destructive operations. The approval interface is a safety-critical control surface, not a convenience feature. The design borrows three proven patterns:

1. **GitHub PR review** — triage panel with pending count badge, per-item review with context, three-button action (approve/deny/reply-first), and an immutable audit trail.
2. **Linear/Notion** — selective notification discipline (only actionable items, not FYIs), grouped related approvals, accurate badge counts.
3. **Google Docs suggesting mode** — per-item granularity (approve/reject individual suggestions), clear visual distinction between pending/resolved states, traceable edit history.

The Canopy-specific innovation is the **approval side panel** as a persistent, collapsible drawer showing: pending count badge, message preview with full context chain, approve/deny/reply-first actions, auto-approval rules, and a scrollable audit trail.

## 1. Survey of Existing Patterns

### 1.1 GitHub Pull Request Review

**Core UX flow:**
- PR list shows pending review count per PR
- Review page: Conversation tab (chronological) + Files Changed tab (side-by-side diff)
- Review actions: Comment, Approve, Request Changes — three distinct buttons, clear intent
- Inline comments on specific lines (threaded, resolvable)
- Review required gate — can't merge without approval
- CI status integrated: green check / red X visible during review
- Submit review: batch all inline comments + decision into one "review" event

**What Canopy borrows:**
- Three-button action model (approve/deny/reply-first)
- Pending count badge on the approval panel icon
- Batch review: review all pending items, submit decisions together
- Immutable audit trail: every approval action logged with author, timestamp, context

**What Canopy rejects:**
- Side-by-side diff view — Canopy approval is about semantic intent ("delete database"), not code diffs. Context chain preview replaces diff.
- "Request Changes" is renamed to "Reply First" — in Canopy, denying without explanation is hostile. Reply-first means "I need to discuss this before approving."

### 1.2 Linear / Notion Notification & Approval

**Core UX flow (Linear):**
- Inbox icon with badge showing unread count
- Notifications grouped by project/thread — not individual spam
- "Mark read" at project level, not per-notification
- Quick actions from notification: change status, assign
- Only actionable notifications, not FYIs

**Core UX flow (Notion):**
- Comment threads on any block — @mentions trigger notifications
- Comments can be "resolved" (hidden, archived) or left open
- Comment sidebar per page showing all unresolved threads
- Suggestion mode: proposed edits shown inline, accept/reject per suggestion

**What Canopy borrows:**
- Badge count in persistent side panel — always visible, zero is hidden
- Grouping: approvals grouped by thread, not individual messages
- Selective notification: only show items needing YOUR action (owner sees everything; friends see only their own pending approvals)
- "Reply first" equals Notion's comment-before-resolve pattern — discussion before decision

**What Canopy rejects:**
- Linear's "mark read" at project level — Canopy approvals are binary (approve/deny), not read/unread
- Notion's suggestion mode ambiguity (users forget they're in suggesting mode) — Canopy's approval panel is ALWAYS visible when there are pending items, mode is explicit

### 1.3 Google Docs Suggesting Mode

**Core UX flow:**
- Toggle between Editing (direct edits) and Suggesting (proposed edits)
- Suggestions appear inline with colored highlights — green = added, strikethrough = deleted
- Each suggestion: accept (checkmark) or reject (X) button
- Comment threads can be attached to suggestions
- Suggestion history: resolved suggestions are archived but accessible
- Version history: full document revision log

**Pain points (from UX research):**
- Mode confusion: users don't know they're in suggesting mode, edits go as suggestions unintentionally
- Discoverability: suggesting mode is hidden behind a dropdown
- Resolution history: resolved suggestions disappear from view, hard to find later

**What Canopy borrows:**
- Per-item accept/reject granularity — each pending action gets its own approve/deny
- Visual distinction: pending (amber/yellow highlight) vs approved (green) vs denied (red strikethrough)
- Context inline: the message preview shows the full conversation chain leading to the action
- Archive accessible: resolved approvals move to audit trail (scrollable, searchable)

**What Canopy rejects:**
- Mode toggle — Canopy approval is always-on, not a mode you enter/exit. Pending approvals are visible regardless of view.
- Editor-role ambiguity (suggesting vs editing) — Canopy distinguishes: human input (always authoritative) vs agent-proposed actions (always require approval)

### 1.4 Maker/Checker Enterprise UX Patterns

**Core UX flow (from UXPin and enterprise best practices):**
- Maker submits, Checker reviews — separation of duties enforced in UI (maker can't approve own requests)
- Pending queue: list view with status indicators, sortable by date/priority/type
- Side-by-side review: original vs proposed change
- Clear actions: approve/reject buttons prominent; rejection requires mandatory comment
- Audit trail: permanent, read-only log of all actions with maker, checker, timestamp, comments, outcome

**What Canopy borrows:**
- Mandatory rejection notes: if owner denies an agent action, they must say why (helps agent learn)
- Separation of duties: if Canopy has multiple human members, the person who invites the agent can't be the sole approver (future: per-tree approval policy)
- Audit trail: immutable `approval_events` table, every action logged
- Queue sorting: sort pending approvals by recency, urgency (agent explicitly flags "this is urgent"), or thread

**What Canopy rejects:**
- Side-by-side comparison — Canopy actions are natural language, not structured form fields. Context chain preview is the comparison mechanism.
- Multi-step maker/checker with routing — Canopy MVP is single-user, so owner is the only approver. Multi-step routing (chain of approvers) is post-MVP.

## 2. Canopy Approval Side Panel — Design Specification

### 2.1 Panel Layout

```
┌──────────────────────────────────────────────┐
│  🔔 Pending (3)                    [Expand]  │  ← Badge with count
│──────────────────────────────────────────────│
│  Thread: Database Migration        [Approve] │
│  Agent: "DROP TABLE users CASCADE"  [Deny]   │
│  Context: Jane said "clean up old   [Reply]  │
│  tables" → agent interpreted as               │
│  DROP production table                        │
│                                              │
│  ─────────────────────────────────────────── │
│                                              │
│  Thread: Weekly Report             [Approve] │
│  Agent: "Sent report to team@..."  [Deny]    │
│  Context: Owner requested weekly   [Reply]   │
│  report generation                            │
│                                              │
│  ─────────────────────────────────────────── │
│                                              │
│  Thread: Config Change             [Approve] │
│  Agent: "Updated max_connections"  [Deny]    │
│  Context: Post-incident review →  [Reply]   │
│  agent proposed config change                 │
│                                              │
│  [Auto-approval rules...]                    │
│  [View audit trail →]                        │
└──────────────────────────────────────────────┘
```

### 2.2 Component Breakdown

#### 2.2.1 Pending Count Badge

- **Location:** Approval panel toggle button (sidebar or top bar)
- **Visual:** Pill badge with count number, amber/yellow fill when count > 0
- **Behavior:** 
  - Hidden when count = 0 (no pending approvals)
  - Animated pulse on new arrival (subtle, non-distracting)
  - Count includes all pending items across all threads
  - Clicking opens/expands the side panel

#### 2.2.2 Pending Item Card

Each pending approval is a card in the scrollable panel:

| Element | Description |
|---------|-------------|
| **Thread label** | Which conversation/thread the action is in |
| **Agent action preview** | First line of what the agent wants to do (bold, prominent) |
| **Context chain** | Expandable: last N messages leading to this action, with author labels |
| **Approve button** | Green/primary — immediate execution |
| **Deny button** | Red/destructive — opens mandatory comment dialog |
| **Reply First button** | Neutral — opens inline reply composer, action stays pending |

#### 2.2.3 Approve Flow

1. Owner clicks "Approve" on a pending item
2. Item slides out with a green checkmark animation (200ms)
3. Agent is notified: "Owner approved: DROP TABLE users CASCADE" — agent executes
4. Item moves to audit trail with status "approved"
5. If this was the last pending item, badge disappears

#### 2.2.4 Deny Flow

1. Owner clicks "Deny" on a pending item
2. Modal dialog: "Why are you denying this? (required)" — text area with submit
3. Owner writes reason: "Don't drop production — I meant the staging tables"
4. Item slides out with red X animation
5. Agent is notified with the denial reason — agent learns and adjusts
6. Item moves to audit trail with status "denied" + reason

#### 2.2.5 Reply First Flow

1. Owner clicks "Reply First" on a pending item
2. Inline composer opens directly below the item card
3. Owner types clarification: "Which tables specifically? I meant old migration tables in staging."
4. Agent receives the reply as a new message in the thread
5. The pending item remains pending — no decision yet
6. Agent may re-propose a modified action based on the clarification

### 2.3 Auto-Approval Rules

Accessible from the bottom of the pending list:

```
┌──────────────────────────────────────────────┐
│  Auto-Approval Rules                         │
│──────────────────────────────────────────────│
│  ✓ Always approve in thread "Deploy Logs"   │  [Edit] [Delete]
│  ✓ Always approve actions from Jane          │  [Edit] [Delete]
│  + Add Rule...                               │
└──────────────────────────────────────────────┘
```

**Rule model:**
- **Scope:** per-thread, per-user (friend), per-action-type (future)
- **Conflict resolution:** most specific rule wins
  - "Always approve Jane" + "Never approve database changes in prod-thread"
  - Jane proposes DROP TABLE in prod-thread → DENIED (thread rule is more specific)
- **Rule matching is server-side:** agent checks rules before even surfacing to approval panel
- **Rule creation:** from the approval panel, or from "Always approve X" on any pending item

### 2.4 Audit Trail

Scrollable, searchable history of all approval decisions:

```
┌──────────────────────────────────────────────┐
│  Audit Trail                          [Filter]│
│──────────────────────────────────────────────│
│  ✅ 2026-07-20 14:32 | Thread: Deploy Logs   │
│     Agent: "Deployed v1.4.2 to staging"      │
│     Approved by: Owner | Auto-rule: "Always   │
│     approve in Deploy Logs thread"            │
│                                              │
│  ❌ 2026-07-20 13:15 | Thread: DB Migration  │
│     Agent: "DROP TABLE users CASCADE"         │
│     Denied by: Owner                          │
│     Reason: "Don't drop production tables —   │
│     I meant staging migration tables"          │
│                                              │
│  💬 2026-07-20 12:40 | Thread: Config Change │
│     Agent: "Updated max_connections to 500"   │
│     Approved by: Owner                        │
│                                              │
│  [Load more...]                              │
└──────────────────────────────────────────────┘
```

**Audit trail properties:**
- Filterable: by thread, by decision (approved/denied), by date range
- Searchable: full-text search over agent actions + denial reasons
- Immutable: once logged, cannot be deleted or modified
- Exportable: CSV/JSON for compliance (post-MVP)
- Visible to: owner always; friends see only their own initiated actions

## 3. Interaction Design

### 3.1 States

| State | Visual | Action Available |
|-------|--------|-----------------|
| **Pending** | Amber left-border, subtle pulse animation on new | Approve / Deny / Reply First |
| **Approving** | Brief spinner on approve button (200ms) | None — in progress |
| **Denying** | Modal dialog open for required comment | Submit / Cancel |
| **Approved** | Green checkmark, slides away (200ms), moves to audit | None — completed |
| **Denied** | Red X, slides away with reason, moves to audit | None — completed |
| **Expired** | Gray, auto-denied after N days (configurable) | None — treated as denied |
| **Conflict** | Amber warning: "Another rule overrides" | Resolve manually |

### 3.2 Keyboard Navigation

| Key | Action |
|-----|--------|
| `Ctrl+Shift+A` | Open approval panel, focus first pending item |
| `j` | Next pending item |
| `k` | Previous pending item |
| `a` | Approve focused item |
| `d` | Deny focused item (opens reason dialog) |
| `r` | Reply First on focused item (opens inline composer) |
| `Esc` | Close panel / close deny dialog |

### 3.3 Mobile Adaptation

- Side panel becomes a bottom sheet (half-height, swipeable)
- Swipe right on pending item = Approve (green reveal)
- Swipe left on pending item = Deny (red reveal, opens reason input)
- Long press = Reply First
- Badge appears in the navigation bar when pending > 0

## 4. Data Model Impact

The approval UX implies the following data structures (to be formalized in Phase 2 SPEC-DM-03):

```
approvals table:
  id: UUIDv7
  tree_id: UUID
  node_id: UUID (the agent's proposed action node)
  thread_id: UUID (which conversation)
  status: pending | approved | denied | expired
  approved_by: UUID (user who approved)
  denied_reason: text (mandatory on deny)
  auto_rule_id: UUID (null if manually approved)
  created_at: timestamp
  decided_at: timestamp

approval_rules table:
  id: UUIDv7
  scope_type: thread | user | action_type
  scope_target: UUID (thread_id or user_id)
  decision: approve | deny
  priority: int (higher = more specific, wins conflicts)
  created_by: UUID
  created_at: timestamp
  active: boolean

audit_log table:
  id: UUIDv7
  approval_id: UUID
  action: pending | approved | denied | expired | rule_created | rule_deleted
  actor: UUID (who took the action)
  details: JSONB (full context at time of action)
  created_at: timestamp (immutable)
```

## 6. Comparison Matrix — Patterns vs Canopy

| Pattern Source | Key Feature | Canopy Adoption | Canopy Adaptation |
|---------------|-------------|-----------------|-------------------|
| GitHub PR | Three-button review | ✓ Approve/Deny/Reply | "Request Changes" → "Reply First" |
| GitHub PR | Batch review submit | ✓ Batch decisions | All pending items can be batch-approved |
| GitHub PR | Inline comments | ✓ Reply First opens inline composer | Threaded discussion before decision |
| Linear | Badge count only for actionables | ✓ Accuracy critical | Zero pending = badge hidden |
| Linear | Grouped notifications | ✓ Grouped by thread | Not per-message spam |
| Notion | Comment before resolve | ✓ Reply First | Discussion before binary decision |
| Notion | Resolved comments archive | ✓ Audit trail | Searchable, filterable, exportable |
| Google Docs | Per-suggestion accept/reject | ✓ Per-item granularity | Each agent proposal = one item |
| Google Docs | Mode confusion (pain point) | ✗ Avoided | Always-on panel, no mode toggle |
| Maker/Checker | Mandatory rejection notes | ✓ Required on deny | Agent learns from rejection reasons |
| Maker/Checker | Separation of duties | ✓ Post-MVP | Multi-user approval policies |
| Maker/Checker | Permanent audit trail | ✓ Immutable | approval_events table, never deleted |
| Enterprise | Status labels (pending/approved/denied) | ✓ Clear state machine | Visual + data model |
| Enterprise | Expiration/aging | ✓ Auto-deny after N days | Configurable per tree |

## 7. Accessibility Considerations

- All approval actions keyboard-accessible (see keyboard shortcuts above)
- ARIA: `role="alert"` for new pending items, `aria-live="polite"` for status changes
- Screen reader: badge count announced as "3 pending approvals"
- Color is not the only indicator: each state has icon + text label + color
- High-contrast mode: borders and icons remain visible without color
- Focus management: after approving an item, focus moves to next pending item (not lost)

## 8. Open Questions for Phase 2

1. **Expiration:** What is the default auto-deny period? (Suggested: 7 days for normal, 24h for agent-flagged "urgent")
2. **Batch approval UX:** Should batch approve have a confirmation dialog? (Suggested: yes, with a count: "Approve all 3 pending actions?")
3. **Denial templates:** Should owners be able to save common denial reasons? (Suggested: post-MVP)
4. **Friend visibility:** Can friends see the owner's approval queue? (Suggested: no — only their own initiated actions)
5. **Agent learning from denials:** How does the agent receive and incorporate denial reasons? (Suggested: denial reason is injected into the agent's context for the next turn in that thread, prefixed with "[Owner denied previous action: ...]")
6. **Multi-user approval chains:** Post-MVP: two-person approval requirement for destructive actions? (Suggested: out of scope for MVP, design the schema to support it)
