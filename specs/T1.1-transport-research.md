# T1.1 — Transport Research: SSE vs WebSocket vs NATS

> Status: COMPLETE | Date: 2026-07-19 | Author: coding-hermes-foreman

## Executive Summary

**Recommendation: SSE as primary transport, NATS as backend message queue, WebSocket as optional bidirectional fallback.**

SSE wins for Hermes Canopy's core use case (server→client tree delta sync) due to built-in reconnection, HTTP/2 multiplexing eliminating connection limits, and Last-Event-ID for lossless resume. NATS complements as a backend pub/sub layer for reliable message delivery and offline queuing. WebSocket adds operational complexity (sticky sessions, manual reconnection) with no benefit for unidirectional sync.

---

## 1. SSE (Server-Sent Events)

### Overview
- Standard: W3C EventSource API, RFC 6202
- Transport: HTTP/1.1 or HTTP/2
- Direction: Unidirectional (server → client)
- Data format: Text (text/event-stream), each event is `data:` lines terminated by `\n\n`
- Native browser API: `new EventSource(url)` — zero dependencies

### Connection Limits

| Protocol | Limit per domain | Notes |
|----------|-----------------|-------|
| HTTP/1.1 | 6 connections | Shared across all tabs. 7th connection blocks. |
| HTTP/2 | ~100 streams (server-configurable) | Multiplexed over single TCP connection. Effectively unlimited for practical use. |

**Decision:** Canopy MUST serve SSE over HTTP/2 (or HTTP/3). This eliminates the connection-limit concern entirely. Go's `net/http` supports HTTP/2 via `golang.org/x/net/http2`. TLS is a prerequisite (h2 requires TLS in browsers, though h2c works in Go). All deployment modes (local, tunnel, relay) get HTTP/2 automatically with a TLS certificate (via Let's Encrypt or tunnel-provided HTTPS).

### Reconnection Behavior
- **Built into browser.** EventSource automatically reconnects on connection loss.
- Reconnection delay: starts at ~3 seconds, browser-implemented exponential backoff.
- `Last-Event-ID` header: sent automatically on reconnect. Server reads this header and resumes the event stream from where the client left off.
- No application code needed for reconnection — the browser handles it.

### Last-Event-ID Behavior Across Browsers

| Browser | Last-Event-ID | Reconnect Delay | Notes |
|---------|---------------|-----------------|-------|
| Chrome 120+ | Sent as HTTP header | ~3s initial, exponential | Standard behavior |
| Firefox 120+ | Sent as HTTP header | ~3s initial, exponential | Standard behavior |
| Safari 17+ | Sent as HTTP header | ~3s initial, exponential | Standard behavior |
| Edge 120+ | Sent as HTTP header | ~3s initial, exponential | Chromium-based, same as Chrome |

All modern browsers implement Last-Event-ID identically. No polyfill needed for EventSource — it's available in all evergreen browsers since 2013.

### Reconnection Backoff Strategy (Server-Side)
The browser retries automatically, but the server can control backoff via the `retry:` field:
```
retry: 5000   # browser waits 5s before reconnecting
```
Server-side strategy for Canopy:
- Default: 3s (browser default)
- After 3rd reconnect in 60s: increase to 10s
- After 10th reconnect: increase to 30s
- Cap at 60s
- Reset to 3s after 2min of stable connection

### Polyfills
**None needed.** EventSource is supported in all browsers Canopy targets:
- Chrome 6+ (2010)
- Firefox 6+ (2011)
- Safari 5+ (2010)
- Edge 79+ (2020, Chromium)
- iOS Safari 4.2+ (2010)
- Android Chrome/WebView 4.4+ (2013)

If polyfill is needed for legacy browsers: `event-source-polyfill` (npm, 2KB gzipped) adds EventSource to IE/old Edge.

## 2. WebSocket

### Overview
- Standard: RFC 6455
- Transport: ws:// (plain) or wss:// (TLS)
- Direction: Full-duplex (bidirectional)
- Data format: Text frames + binary frames
- No native reconnection API

### Sticky Session Requirements
Horizontal scaling of WebSocket requires sticky sessions:

| Load Balancer | Sticky Session Method | Notes |
|---------------|----------------------|-------|
| Nginx | `ip_hash` or `sticky` (plus module) | ip_hash breaks on mobile IP rotation |
| HAProxy | `stick-table` with cookie | More reliable than IP-based |
| AWS ALB | Application-based stickiness (cookie) | AWS-managed, auto-handles |
| Kubernetes | Session Affinity via Service | `sessionAffinity: ClientIP` |
| Traefik | Sticky sessions via cookie | Built-in |

**Without sticky sessions:** WebSocket handshake lands on server A, subsequent frames may hit server B (which has no connection state). Requires pub/sub backplane (Redis, NATS, Kafka) to broadcast frames to all servers. Adds latency and infrastructure complexity.

**Recommendation for Canopy:** If WebSocket is ever needed (collaborative editing), use a NATS backplane. NATS is already in the stack for message queuing — WebSocket frames fan out through NATS subjects to all backend instances. This eliminates the need for sticky sessions entirely.

### Reconnection
- **No built-in support.** Application must implement:
  - Exponential backoff (1s → 2s → 4s → 8s → 16s → 30s max)
  - Jitter (±25%) to prevent thundering herd
  - State resynchronization after reconnect (server may have pushed data while disconnected)
- Typical implementation: 50-100 lines of JavaScript for a robust reconnecting WebSocket wrapper.
- Libraries exist: `reconnecting-websocket` (npm, unmaintained), `robust-websocket` (npm).

## 3. NATS

### Overview
- Type: Message queue / pub-sub system (not a browser transport)
- Protocol: NATS protocol over TCP or WebSocket
- Server: `nats-server` (Go, single binary, 12MB)
- Performance: 10M+ messages/sec, sub-millisecond latency (backend)
- Persistence: JetStream (at-least-once delivery, replay, consumer groups)

### Browser Connectivity
NATS itself is not a browser protocol. Browsers connect via:
1. **NATS WebSocket** (nats.ws): NATS server exposes WebSocket endpoint. Browser connects via `nats.ws` JavaScript client. Communication is NATS protocol over WebSocket frames.
2. **SSE Gateway**: Go backend subscribes to NATS subjects, translates to SSE events for browser clients. Cleaner separation — browser only speaks SSE.

**Recommended pattern for Canopy:**
```
Browser (EventSource) ←→ Go Backend ←→ NATS ←→ Go Backend ←→ Browser
```
NATS is the backend message fabric. Browsers never speak NATS directly. The Go gateway translates between SSE (browser-facing) and NATS (backend pub/sub).

### Why NATS Complements SSE
- **Offline queuing:** When a browser disconnects, NATS JetStream buffers events for that client. On reconnect, the Go gateway replays buffered events.
- **Multi-server fan-out:** Tree update from any backend instance → publish to NATS subject → all backend instances receive → each pushes to its connected SSE clients.
- **Decoupling:** SSE handles browser transport. NATS handles server-to-server communication. Each does what it's best at.
- **Operational simplicity:** Single `nats-server` binary, no cluster needed for single-node. Embedded mode available via `nats-server -js` (JetStream included).

## 4. Comparison Matrix

| Feature | SSE | WebSocket | NATS |
|---------|-----|-----------|------|
| Direction | Server→Client | Bidirectional | Pub/Sub (backend) |
| Browser API | `EventSource` (native) | `WebSocket` (native) | `nats.ws` (library) |
| Reconnection | Built-in + Last-Event-ID | Manual (50-100 LOC) | N/A (backend) |
| Connection limit (HTTP/1.1) | 6 per domain | 6 per domain | N/A |
| Connection limit (HTTP/2) | ~100 streams (no practical limit) | Same | N/A |
| Sticky sessions needed | No | Yes (without backplane) | No (pub/sub) |
| Binary data | Base64 only | Native binary frames | Native |
| Browser support | All modern (2010+) | All modern (2010+) | Via nats.ws |
| Implementation complexity | Low | Medium | High (infra) |
| Go server support | `net/http` (stdlib) | `gorilla/websocket` or `nhooyr.io/websocket` | `nats.go` |
| Offline queue | None (server must implement) | None (server must implement) | JetStream (built-in) |
| Multi-server | Stateless, any server can reply | Stateful, needs pub/sub | Pub/sub native |

## 5. Decision & Rationale

### Primary Transport: SSE (HTTP/2)
**Why SSE over WebSocket:**
1. Tree sync is server→client: client requests changes via REST POST, server pushes deltas via SSE. Bidirectional WebSocket adds complexity with no benefit.
2. Reconnection is free: EventSource + Last-Event-ID gives lossless resume. WebSocket requires manual implementation.
3. HTTP/2 eliminates the 6-connection limit: practical limit is 100+ streams per domain — more than Canopy will ever need per user.
4. No sticky sessions: SSE is stateless HTTP. Any backend instance can serve any client's SSE stream (events come from NATS). Load balancing is trivial (round-robin).
5. Infrastructure simplicity: standard HTTP, works through all proxies, no special LB config, no ws:// upgrade headaches.
6. Go stdlib: `net/http` handles SSE natively — `Flusher` interface + `Content-Type: text/event-stream`.

### Backend Message Layer: NATS
**Why NATS:**
1. Offline queuing: JetStream buffers events when client disconnects, replays on reconnect.
2. Multi-server fan-out: tree updates published to NATS → all Go instances receive → each pushes to its SSE clients.
3. Decoupling: transport (SSE) and messaging (NATS) are separate concerns.
4. Single binary: `nats-server` is 12MB, runs embedded in tests, production is standalone.
5. Go-native: `nats.go` client is first-class.

### When WebSocket Would Be Needed
Only if Canopy adds **collaborative real-time editing** (two users editing the same node simultaneously, like Google Docs). The bidirectional nature of WebSocket enables operational transform / CRDT sync with sub-100ms latency. This is a Phase 5+ concern. SSE handles everything through Phase 4.

## 6. Recommended Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                        BROWSER                              │
│  ┌──────────────────────┐    ┌──────────────────────────┐   │
│  │   EventSource (SSE)  │    │   fetch() / REST (POST)  │   │
│  │   GET /trees/X/events│    │   POST /trees/X/nodes    │   │
│  └──────────┬───────────┘    └────────────┬─────────────┘   │
└─────────────┼─────────────────────────────┼─────────────────┘
              │ HTTPS (HTTP/2)              │ HTTPS
              ▼                             ▼
┌─────────────────────────────────────────────────────────────┐
│                    Go Backend (canopyd)                     │
│  ┌──────────────────────┐    ┌──────────────────────────┐   │
│  │   SSE Hub            │    │   REST Handlers          │   │
│  │   - Connection mgr   │    │   - Tree CRUD            │   │
│  │   - Last-Event-ID    │    │   - Node CRUD            │   │
│  │   - Heartbeat        │    │   - Approval             │   │
│  └──────────┬───────────┘    └────────────┬─────────────┘   │
│             │                             │                 │
│             ▼                             ▼                 │
│  ┌──────────────────────────────────────────────────────┐   │
│  │               NATS Client (nats.go)                  │   │
│  │  Pub: tree.<id>.events   Sub: tree.<id>.commands     │   │
│  └──────────────────────┬───────────────────────────────┘   │
└─────────────────────────┼───────────────────────────────────┘
                          │ NATS Protocol
                          ▼
              ┌───────────────────────┐
              │   NATS Server         │
              │   + JetStream         │
              │   (offline queue)     │
              └───────────────────────┘
```

### Event Flow
1. User creates node → REST POST to Go backend
2. Go backend validates, persists to PostgreSQL, publishes to NATS `tree.<id>.events`
3. All Go instances subscribed to `tree.<id>.events` receive the event
4. Each instance pushes the event to its connected SSE clients for that tree
5. If client is offline: JetStream buffers the event, replays on reconnect (Last-Event-ID resume)
