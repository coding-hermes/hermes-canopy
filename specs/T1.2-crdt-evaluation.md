# T1.2 — CRDT Library Evaluation: Yjs vs Automerge

> Status: COMPLETE | Date: 2026-07-20 | Author: coding-hermes-foreman

## Executive Summary

**Recommendation: Yjs for Canopy's tree data model.**

Yjs wins decisively for Canopy's use case: 18KB gzipped (vs 320KB+ WASM), pure JavaScript with zero initialization cost, the largest CRDT ecosystem (920K weekly downloads, 17K stars), first-class nested Map/Array types that map directly to tree structures, and built-in offline support via y-indexeddb. Automerge 3.0 has made impressive strides (10x memory reduction, 17h→9s load times), but its 35.9MB install size and ~50ms WASM startup cost are unnecessary overhead for Canopy's tree sync model where branching naturally avoids conflicts.

---

## 1. Library Overview

### Yjs
- **Repository:** github.com/yjs/yjs
- **Language:** Pure JavaScript (no native deps, no WASM)
- **License:** MIT
- **Stars:** ~17,000
- **Weekly Downloads:** ~920,000
- **Latest Version:** 13.6.x (stable)
- **Core Concept:** Exposes CRDT internals as shared data types (Y.Map, Y.Array, Y.Text, Y.XmlFragment). Changes are automatically distributed to peers and merged without conflicts.

### Automerge
- **Repository:** github.com/automerge/automerge
- **Language:** Rust core + JavaScript bindings via WASM
- **License:** MIT
- **Stars:** ~6,400
- **Weekly Downloads:** ~85,000 (@automerge/automerge)
- **Latest Version:** 3.0.x (stable, released 2025)
- **Core Concept:** JSON-like CRDT — documents are JSON objects. Concurrent modifications merge automatically. Full DAG of editing history (Git-like change tracking).

## 3. Data Model Fit for Tree Structures

### Canopy's Tree Data Model (from DuckBrain)
```
Tree
├── Node (root)
│   ├── Node (reply)
│   │   └── Node (reply)
│   └── Node (branch/fork)
│       ├── Node (reply)
│       └── Node (reply)
└── Node (sibling branch)
```

Each node has: id (UUIDv7), content, author, created_at, edited_at, approval_status, metadata. Nodes have parent-child relationships (edges). Branches are created by replying to any node (creates "reply" edge) or forking from a node (creates "branch" edge).

### Yjs: Nested Y.Map + Y.Array

```typescript
const ydoc = new Y.Doc()
const treeMap = ydoc.getMap('tree')

// Each node is a Y.Map
const nodeMap = new Y.Map()
nodeMap.set('id', 'uuid-1')
nodeMap.set('content', 'Hello world')
nodeMap.set('author', 'user-1')
nodeMap.set('createdAt', Date.now())

// Children stored as Y.Array (ordered) on each node
const children = new Y.Array()
children.push([childNodeMap1, childNodeMap2])

nodeMap.set('children', children)
treeMap.set('uuid-1', nodeMap)
```

**How branching works:**
- Replying to node A creates a new Y.Map, pushes it to A's `children` Y.Array
- Forking from node A creates a new Y.Map, pushes it to A's `children` Y.Array with a `branch` flag
- Tree traversal: recursively walk Y.Maps through their `children` Y.Arrays
- Node lookup by UUID: `treeMap.get(uuid)` — O(1) direct access
- IndexedDB persistence: `y-indexeddb` provider stores the entire Y.Doc

**Conflict resolution for concurrent branch creation:**
- Two users concurrently replying to the same node: both replies are appended to the `children` Y.Array. Yjs uses LSEQ (Linear Sequence) CRDT for array insertion — both children appear in the array in a deterministic order based on their Lamport timestamps. No data loss. No conflict dialog needed.
- Two users concurrently editing the SAME node's `content`: Y.Map uses LWW (last-write-wins) per key. The later timestamp wins. The earlier value is lost.
- Two users editing DIFFERENT properties of the same node (one edits `content`, other edits `approval_status`): both changes survive — different keys, no conflict.

### Automerge: JSON CRDT

```typescript
import * as Automerge from '@automerge/automerge'

let doc = Automerge.init()
doc = Automerge.change(doc, (d) => {
  d.tree = {}
  d.tree['uuid-1'] = {
    id: 'uuid-1',
    content: 'Hello world',
    author: 'user-1',
    createdAt: Date.now(),
    children: []
  }
})
```

**How branching works:**
- Replying to node A: push new node object to `doc.tree['uuid-A'].children` array
- Forking from node A: same mechanism, different edge type metadata
- Tree traversal: standard object/array access — `doc.tree[id].children.forEach(...)`
- Node lookup by UUID: `doc.tree[uuid]` — direct object key access

**Conflict resolution for concurrent branch creation:**
- Two users concurrently pushing to the same `children` array: Automerge's list CRDT merges both additions. Order is deterministic (based on operation IDs). Both children appear.
- Two users concurrently editing the SAME key on the same node: Automerge picks a single deterministic winner (based on Lamport timestamps + actor ID). The losing value is retained internally in the operation history but not visible in the document state — you'd need to inspect the change DAG to recover it.
- Two users editing DIFFERENT keys: both changes survive.

### Data Model Comparison

| Aspect | Yjs | Automerge | Winner |
|--------|-----|-----------|--------|
| Tree = nested maps | Natural: `Y.Map` nesting | Natural: JSON object nesting | Tie |
| Ordered children | `Y.Array` with LSEQ CRDT | `Array` with list CRDT | Tie |
| Node lookup by UUID | `treeMap.get(uuid)` O(1) | `doc.tree[uuid]` O(1) | Tie |
| Concurrent branch creation | Both survive, deterministic order | Both survive, deterministic order | Tie |
| Concurrent same-property edit | LWW, earlier lost | Single winner, earlier retained in history | Automerge |
| Type safety | Weak (any JSON-serializable) | Weak (any JSON-serializable) | Tie |
| Schema enforcement | Manual validation | Manual validation | Tie |

## 4. Memory Usage

| Document Size | Yjs (estimated) | Automerge 3.0 (estimated) | Notes |
|---------------|-----------------|--------------------------|-------|
| 100 nodes | <1 MB | <1 MB | Both trivial |
| 1,000 nodes | ~5-10 MB | ~2-5 MB | Automerge 3.0's columnar compression is more memory-efficient |
| 10,000 nodes | ~50-100 MB | ~10-20 MB | Yjs stores full operation log; Automerge compresses |
| 100,000 nodes | ~500MB+ (may degrade) | ~100-200 MB | Yjs GC limited for 100K+ operations |

**Automerge 3.0 memory improvements:**
- 10x reduction over Automerge 2.0 (100x in some cases)
- On-disk columnar compression format now used in-memory
- Documents that took 17 hours to load in 2.0 now load in 9 seconds
- Backward compatible with Automerge 2.0 documents

**Yjs memory characteristics:**
- Stores full operation log (structs) for CRDT merge
- Version snapshots require additional Version Vector + Delete Set storage
- Tombstone accumulation: deleted Y.Map entries leak memory (known issue, yjs#741)
- Garbage collection is limited — large docs (100K+ operations) degrade
- y-indexeddb provider offloads updates to IndexedDB, reducing in-memory pressure

**Canopy projection:** A typical Hermes conversation tree has hundreds to low thousands of nodes. At 10K nodes, Yjs at ~50-100MB is acceptable for a desktop WebUI app. Automerge 3.0's 10-20MB is better for memory-constrained mobile use but Canopy is desktop-first.

## 6. Sync & Transport Integration

### Yjs Sync Protocol

Yjs uses a binary update protocol:
1. `Y.encodeStateAsUpdate(ydoc)` → binary Uint8Array of all changes
2. `Y.applyUpdate(ydoc, update)` → merge remote changes
3. Update exchange: peers send full state or incremental diffs

For SSE integration:
```typescript
// Client sends local changes via REST POST
const update = Y.encodeStateAsUpdate(ydoc, Y.encodeStateVector(ydoc))
await fetch('/trees/X/sync', { method: 'POST', body: update })

// Server pushes updates via SSE
const eventSource = new EventSource('/trees/X/events')
eventSource.onmessage = (e) => {
  const update = base64ToUint8Array(e.data)
  Y.applyUpdate(ydoc, update)
}
```

### Automerge Sync Protocol

Automerge uses a document-level sync protocol via `automerge-repo`:
1. `repo.find(docId)` → returns handle to document
2. `handle.change((doc) => { ... })` → modify
3. Repo handles networking, storage, sync automatically

For SSE integration, less direct — Automerge Repo abstracts networking behind adapters. An SSE adapter would need to be custom-built (no existing `@automerge/*-network-sse` package exists).

**Sync integration verdict:** Yjs's raw binary update protocol is simpler to bridge to a custom SSE transport. Automerge Repo abstracts networking but expects pluggable adapters — an SSE adapter is custom work either way.

## 8. Canopy-Specific Scenarios

### Scenario A: Two users concurrently reply to same node
This is the primary conflict scenario for Canopy.

**Yjs:** Both replies are appended to the `children` Y.Array. LSEQ CRDT ensures deterministic ordering across all peers. Both replies appear. No data loss. No UI conflict dialog needed.

**Automerge:** Both replies are pushed to the `children` array. List CRDT merges both. Both appear in deterministic order. No data loss. No UI conflict dialog needed.

**Verdict:** Both handle this correctly. Tie.

### Scenario B: Two users concurrently edit same node's content
Less common but possible (rapid edits, offline→online).

**Yjs:** Y.Map uses LWW on the `content` key. Last writer wins. Earlier content is lost — no history retained (unless you implement app-level versioning).

**Automerge:** Picks a single deterministic winner. Losing value is retained in the document's internal change DAG, recoverable via `Automerge.getHistory()`. This is a meaningful advantage for an approval-based system where audit trails matter.

**Verdict:** Automerge wins for audit trail preservation. Yjs would need app-level versioning.

### Scenario C: Offline→online sync after 50 edits
User writes 50 messages offline, then reconnects.

**Yjs:** `y-indexeddb` persists all 50 operations locally. On reconnect, Yjs syncs the incremental update (only new operations since last sync vector). Efficient.

**Automerge:** `automerge-repo-storage-indexeddb` persists locally. On reconnect, Repo syncs changes automatically. `Automerge.getMissingDeps()` figures out what's missing. Efficient.

**Verdict:** Both handle this well. Tie.

### Scenario D: Tree visualization — reactive rendering
The UI needs to re-render on every tree change. React + Yjs/automerge.

**Yjs:** `ymap.observe()` fires granular events: which key changed, old value, new value. React integration via `useMap` or manual observer in useEffect. Can pinpoint exactly which node changed for efficient re-render.

**Automerge:** `Automerge.diff(oldDoc, newDoc)` computes patches between two states. `handle.on('change', ...)` fires on changes. More coarse-grained — the full document snapshot changes. React integration via `@automerge/automerge-repo-react-hooks` provides `useDocument` hook.

**Verdict:** Yjs's granular observe events are better for large-tree rendering where you want to update only the changed branch, not re-render the entire tree.

## 10. Decision & Rationale

### Primary: Yjs

**Why Yjs over Automerge:**

1. **Bundle size matters for PWA/distribution.** At 18KB vs 320KB gzipped, Yjs loads instantly on any connection. Canopy ships as a WebUI native app — fast cold start is a priority. Automerge's 35.9MB install size bloats the build artifact.

2. **No WASM dependency.** Pure JavaScript means no WASM binary to load, no cross-platform WebView compatibility concerns (some WebView versions have spotty WASM support), and no startup latency. Yjs works everywhere `EventSource` works.

3. **Granular reactivity for large trees.** `Y.Map.observe()` fires events that tell you exactly which property changed on which node. For rendering a 500+ node tree, this means you update only the affected branch, not re-render the entire tree. Automerge's diff-based approach re-computes patches between two document snapshots, which is coarser.

4. **Ecosystem maturity.** 920K weekly downloads, 17K stars, 50+ providers. Yjs is the production default for collaborative editing. Battle-tested by companies like Linear, Notion, and Jupyter.

5. **Direct sync protocol.** Yjs's binary update exchange maps cleanly to Canopy's SSE + REST architecture. `Y.encodeStateAsUpdate()` → binary → base64 → SSE event. `Y.applyUpdate()` on the other side. No abstraction layer to fight.

6. **Conflict resolution matches Canopy's model.** For tree branching, concurrent replies to the same node BOTH survive (LSEQ ordering). The only lossy case is concurrent edits to the SAME property of the SAME node — which is rare in Canopy's UX (users reply or branch, they don't edit each other's messages). And even then, LWW is acceptable for a chat/collaboration surface (the message that arrived later wins).

### When Automerge Would Be Preferred

1. **Audit trail is a hard requirement.** If Canopy needs to prove "what did the document look like at time T?" for compliance/legal reasons, Automerge's built-in Git-like change DAG is invaluable. Yjs would require building app-level versioning on top.

2. **10K+ node trees are common.** Automerge 3.0's memory efficiency (10-20MB vs 50-100MB for 10K nodes) matters if Canopy routinely handles very large trees. For typical Hermes conversations (hundreds to low thousands of nodes), Yjs is fine.

3. **Multi-format document support.** If Canopy expands beyond text messages to rich documents (tables, counters, nested rich-text), Automerge's Text CRDT + Table CRDT + Counter CRDT cover more ground natively.

## 11. Recommended Architecture

```
┌─────────────────────────────────────────────────────┐
│                   BROWSER (WebUI)                    │
│                                                     │
│  ┌──────────────────────────────────────────────┐   │
│  │           React Tree UI                       │   │
│  │  - observe() events drive re-render           │   │
│  │  - Y.Map.get(uuid) for node lookup            │   │
│  │  - Y.Array for ordered children                │   │
│  └──────────────────┬───────────────────────────┘   │
│                     │                               │
│  ┌──────────────────▼───────────────────────────┐   │
│  │           Y.Doc (single document)             │   │
│  │  treeMap: Y.Map<uuid, Y.Map<Node>>            │   │
│  │    └─ Y.Map<Node>                             │   │
│  │         ├─ id: string                         │   │
│  │         ├─ content: string                    │   │
│  │         ├─ author: string                     │   │
│  │         ├─ createdAt: number                  │   │
│  │         ├─ approvalStatus: string             │   │
│  │         └─ children: Y.Array<Y.Map<Node>>     │   │
│  └──────┬──────────────────┬────────────────────┘   │
│         │                  │                         │
│  ┌──────▼──────┐   ┌──────▼──────────────────────┐  │
│  │ y-indexeddb │   │  Custom SSE Provider         │  │
│  │ (offline)   │   │  - Y.encodeStateAsUpdate()   │  │
│  │             │   │  - POST /trees/X/sync         │  │
│  │             │   │  - EventSource /trees/X/events│  │
│  └─────────────┘   └─────────────────────────────┘  │
│                                                     │
└─────────────────────────────────────────────────────┘

  y-indexeddb: auto-persists every Y.Doc update to IndexedDB
  SSE Provider: subscribes to EventSource, Y.applyUpdate() on each event
  REST POST: sends local changes as binary updates
  Go Backend: receives updates, publishes to NATS, fans out to SSE clients
               (see T1.1 for transport architecture)
```

### Data Schema (Canonical Y.Doc shape)

```typescript
// Top level: one Y.Doc per tree
const ydoc = new Y.Doc()

// Tree root map — keyed by node UUID
const treeMap: Y.Map<Y.Map> = ydoc.getMap('tree')

// Node shape (as Y.Map entries)
interface NodeShape {
  id: string           // UUIDv7
  content: string      // message body (markdown)
  author: string       // user ID or profile ID
  createdAt: number    // Unix ms
  editedAt: number     // Unix ms or 0
  approvalStatus: 'pending' | 'approved' | 'denied' | 'none'
  edgeType: 'reply' | 'branch' | 'root'
  metadata: Y.Map      // extensible key-value
  children: Y.Array<Y.Map<NodeShape>>  // ordered child nodes
}

// Usage:
const node = new Y.Map()
node.set('id', '018f3...')
node.set('content', 'Hello world')
node.set('children', new Y.Array())
treeMap.set(node.get('id'), node)
```

## 12. Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Yjs tombstone leak (yjs#741) | Medium | Implement periodic doc compaction: encode as snapshot, create new Y.Doc from snapshot, reload. Document max lifetime ~100K operations before compaction. |
| Yjs memory at 10K+ nodes | Low | Canopy targets desktop WebUI with 50-100MB headroom. If mobile becomes a priority, implement lazy loading (only load visible branch into Y.Doc, fetch rest on demand). |
| LWW data loss on concurrent edit | Low | Canopy UX prevents concurrent edits to same node — only author can edit their own message. Approval changes are atomic (pending→approved). If simultaneous edit becomes a feature, implement app-level versioning on top of Yjs. |
| Custom SSE provider maintenance | Low | The SSE provider is thin (~100 LOC): EventSource → base64 decode → Y.applyUpdate(). No protocol complexity. |
| Automerge 3.0 continues improving | Info | Monitor Automerge adoption. If WASM startup drops below 10ms and ecosystem matures, re-evaluate for Phase 5+. The decision is revokable — switching CRDT libraries is isolated to the Tree Data Store layer (FE-02). |
