## Detailed Evaluation

### 1. React Flow (@xyflow/react v12)

**Architecture:** DOM-based node rendering with a virtual viewport. Each node is a React component. Edges are SVG paths. The viewport applies CSS transforms for pan/zoom.

**Canopy-specific strengths:**
- **Rich message nodes.** Each tree node is a chat message — needs avatars, markdown rendering, code blocks, action buttons. React Flow lets you render full React components as nodes. No foreignObject hacks, no Canvas text measurement math.
- **Keyboard-first navigation.** Built-in props: `nodesFocusable`, `edgesFocusable`, `deleteKeyCode`, `multiSelectionKeyCode`. Arrow keys navigate between connected nodes. Tab through elements. This directly maps to Canopy's j/k/h/l navigation spec.
- **Virtualization.** React Flow uses a viewport-aware render: only nodes in the visible area (plus buffer) are mounted. With proper `React.memo` and `arePropsEqual` selectors, 500+ node trees stay at 60fps.
- **Dark mode.** Built-in CSS variables for theming. Matches Canopy's expected dark theme.
- **SSR support (v12).** TSDoc on every API. Better DX.

**Canopy-specific weaknesses:**
- **No tree layout built-in.** Must pair with dagre (simple), d3-hierarchy (advanced), or elkjs (complex). Tree layout is computed externally and fed as node positions.
- **DOM ceiling.** At ~500 visible DOM nodes, performance degrades. Virtualization mitigates this by only rendering viewport-visible nodes. For Canopy's tree navigation (3-level: macro/branch/merge), the viewport rarely shows >100 nodes simultaneously — well within safe range.
- **No built-in branch collapse/expand.** Must implement custom: hide children on branch collapse, re-run layout, animate positions. This is the main custom work.

**Bundle impact:** ~45KB gzipped. Acceptable. Add dagre (+8KB) or d3-hierarchy (+25KB) for layout = ~53-70KB total. Well under the 500KB Phase 5 budget.

**Accessibility audit:**
- Screen reader: DOM nodes are real elements — headings, buttons, links all work. Add `role="tree"`, `role="treeitem"`, `aria-expanded`, `aria-level`, `aria-selected`.
- Keyboard: Arrow keys navigate between nodes. Tab into node content. Enter to expand/collapse. Escape to zoom out.
- Focus management: `nodesFocusable={true}` enables focus cycling. Must handle focus restoration after layout changes.
- High contrast: CSS variables for theming.
- Reduced motion: disable transition animations via `animationDuration={0}`.

### 2. D3.js + d3-hierarchy

**Architecture:** SVG-based data visualization. `d3.tree()` computes node positions. `d3.zoom()` adds pan/zoom. Data join pattern: enter/update/exit.

**Canopy-specific strengths:**
- **Best tree layout algorithms.** Buchheim's algorithm (d3.tree), Reingold-Tilford "tidy" tree, cluster layout, radial tree. All handle variable node sizes.
- **Animation.** Data join transitions — nodes smoothly animate to new positions on branch/collapse.
- **Mature ecosystem.** 14M weekly downloads. Every edge case solved somewhere.

**Canopy-specific weaknesses:**
- **React integration is painful.** D3 and React both want to own the DOM. Solutions exist (useRef + useEffect, react-d3-tree wrappers) but they fight Canopy's Yjs-powered reactive updates. Every CRDT change → D3 re-compute → enter/update/exit → possible desync with React state.
- **SVG text limitations.** Chat messages need rich text, code blocks, action buttons. SVG `<foreignObject>` is the hack, but it's buggy in Safari, doesn't size correctly, and breaks accessibility. Every rich message requires a foreignObject wrapper.
- **Accessibility is manual.** SVG elements aren't real DOM. Must mirror the entire tree as an invisible ARIA list. Two sources of truth.
- **Performance ceiling.** SVG creates a DOM element per node + per edge. 500 nodes = 500+ SVG group elements. Browser SVG renderer slows down. Canvas beats SVG for large trees.

**Bundle impact:** d3-hierarchy alone is ~25KB gzipped. But practical usage requires d3-selection, d3-zoom, d3-transition → closer to 50KB. Full d3 is ~50KB gzipped.

### 3. Custom Canvas Renderer

**Architecture:** `<canvas>` element with 2D context or WebGL. Spatial index (quadtree/R-tree) for hit testing. Custom event system maps canvas coordinates to nodes.

**Canopy-specific strengths:**
- **Maximum performance.** 5000+ nodes at 60fps. No DOM overhead per node. Perfect for macro view of massive trees.
- **Full rendering control.** Draw anything: custom edge styles (curved, orthogonal, organic), node shapes, branch color coding, merge visualizations.
- **No library churn.** Zero dependency risk for the rendering layer.

**Canopy-specific weaknesses:**
- **No accessibility.** Canvas is a black rectangle to screen readers. Must build a parallel ARIA tree (virtual DOM) that mirrors every node. Every interaction must update both. This is 2-3x the development cost.
- **No text selection.** Canvas text can't be selected, copied, or searched natively. Must implement custom text selection (hit testing, selection handles, clipboard API).
- **No CSS styling.** Every visual change is code. Dark mode? Re-draw with different colors. Font size preference? Re-measure and re-draw. Plugin system (SPEC-PL-01) can't inject CSS.
- **Build everything.** Text layout engine (word wrap, rich text, code blocks). Hit testing. Focus management. Scroll inertia. Pinch zoom. Branch collapse animation. Edge routing. Context menus. Every interaction from scratch.
- **Development cost.** 4-8 weeks for a basic tree render. 12-16 weeks for production quality with accessibility. This delays Canopy by months.

### 4. Cytoscape.js

**Architecture:** Canvas-based graph visualization. Nodes + edges model. Extensible via plugins.

**Canopy-specific strengths:**
- Canvas performance with built-in interactions.
- Mature graph algorithms (centrality, clustering, pathfinding).
- Plugin ecosystem for tree layouts.

**Canopy-specific weaknesses:**
- **Graph-centric, not tree-centric.** Cytoscape's mental model is networks/graphs, not hierarchical trees. Tree layouts are extensions, not first-class. Branch/collapse, ancestor path visualization, and merge views require bending the graph model.
- **Canvas accessibility problem** (same as Custom Canvas).
- **Large bundle** (~120KB gzipped). Heavier than React Flow + d3-hierarchy combined.

**Verdict:** Wrong mental model for a tree-native chat interface.

## Decision

**Primary: React Flow (@xyflow/react v12) for all tree rendering.**

Rationale:
1. **DOM nodes = chat messages.** Canopy nodes are rich content (markdown, code, actions). React components as nodes is a direct 1:1 fit. No foreignObject hacks, no Canvas text math.
2. **Built-in accessibility.** Keyboard nav, ARIA attributes, screen reader support. Canopy's accessibility requirement (SPEC-FE-10) is satisfied out of the box.
3. **React + Yjs integration.** React Flow's `useNodesState` / `useEdgesState` hooks map directly to Yjs Y.Map observations. CRDT change → React Flow update → re-render only changed nodes (via memo).
4. **Performance is sufficient.** The 3-level navigation (macro/branch/merge) limits visible nodes to ~50-200. Well within React Flow's DOM comfort zone. Virtualization handles edge cases.
5. **Zoom/pan + touch built-in.** No custom gesture detection needed.

**Layout engine: d3-hierarchy paired with React Flow.**

React Flow has no built-in tree layout. d3-hierarchy provides the best tree layout algorithms (Buchheim, Reingold-Tilford). Layout is computed as positions (x, y per node), fed to React Flow as `setNodes()`. Layout recomputes on branch/collapse with animated transitions (CSS transitions on node position).

**Fallback: Custom Canvas renderer for extreme scale.**

If Canopy trees grow beyond 2000+ visible nodes (unlikely in current design), implement a Canvas overlay for the macro view only. The branch/merge views stay DOM-based. This is a Phase 7+ optimization, not Phase 5.

## Key Implementation Details

### Virtualization Strategy

React Flow renders only viewport-visible nodes by default. Additional optimization:
- **Macro view:** Show collapsed branch summaries (title + message count). Render 20-50 nodes.
- **Branch view:** Show full message chain from root to leaf. Render 5-100 nodes.
- **Merge view:** Two branches side-by-side. Render 20-80 nodes total.

### Branch Collapse/Expand

1. User collapses branch at node N → mark `collapsed: true` in Yjs
2. d3-hierarchy recomputes layout excluding N's children
3. React Flow `setNodes(newPositions)` with CSS `transition: transform 200ms`
4. Children nodes get `hidden: true` (unmounted from React Flow, not deleted from Yjs)

### Accessibility Map

| WCAG 2.1 Criterion | React Flow Solution |
|---------------------|---------------------|
| 1.3.1 Info and Relationships | `role="tree"`, `role="treeitem"`, `aria-level` |
| 2.1.1 Keyboard | Arrow keys (j/k = prev/next sibling, h/l = parent/child) |
| 2.1.2 No Keyboard Trap | Focus always escapable to chrome |
| 2.4.3 Focus Order | Logical tree traversal order |
| 2.4.7 Focus Visible | Highlighted node border |
| 2.5.5 Target Size | 44x44px touch targets per WCAG |
| 4.1.2 Name, Role, Value | `aria-label` on each node, `aria-expanded` for branches |

### Mobile Touch

React Flow's built-in touch handling:
- Pinch to zoom
- Single-finger pan
- Tap to select node
- Double-tap to drill into branch
- Long-press for context menu
- Swipe to navigate siblings (custom, but easy addition)
