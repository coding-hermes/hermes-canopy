# T1.4 — Offline-Stack Research

> Status: COMPLETE | Date: 2026-07-20 | Author: coding-hermes-foreman

## Executive Summary

**Recommendation: Service Worker (Workbox v7) + y-indexeddb + Custom SSE Provider + Background Sync queue. No SQLite WASM in MVP. Delta Chat as optional extreme-fallback relay (post-MVP).**

Yjs's built-in offline story (y-indexeddb) covers local persistence with zero additional work. A custom SSE-based Yjs provider maps Canopy's existing SSE transport (T1.1 decision) onto Yjs's provider interface, giving us real-time sync when online and local-first operation when offline. Background Sync queues mutations while offline and replays them on reconnect — this is progressive enhancement on top of IndexedDB queuing. SQLite WASM adds 320KB+ WASM payload and Web Worker complexity for no benefit in MVP (CRDTs don't need SQL). Delta Chat is an intriguing fallback but email latency (30s–5min) makes it a last-resort relay, not a primary offline transport.

## 1. Service Worker Caching Strategies

### Strategy Matrix for Canopy

| Asset Type | Strategy | Rationale |
|---|---|---|
| Static assets (JS/CSS chunks, fonts, logos) | **CacheFirst** | Hashed filenames = immutable. Cache once, serve instantly. |
| App shell (index.html, manifest.json) | **StaleWhileRevalidate** | Show cached shell instantly, update in background. |
| SSE event stream | **NetworkFirst** (10s timeout) | Live tree data. Fall back to cached snapshot if offline. |
| API responses (tree CRUD, approvals) | **NetworkFirst** → queue mutation if offline | Mutations go to IndexedDB queue first, Background Sync replays. |
| Card data (File, Task, Code) | **NetworkFirst** with IndexedDB fallback | Cards are structured data, not CRDT. Cache separately. |

### Workbox v7

```js
import { registerRoute } from 'workbox-routing';
import { CacheFirst, NetworkFirst, StaleWhileRevalidate } from 'workbox-strategies';
import { ExpirationPlugin } from 'workbox-expiration';

// Static assets (hashed — immutable)
registerRoute(
  ({ request }) => request.destination === 'script' || request.destination === 'style',
  new CacheFirst({ cacheName: 'static-assets', plugins: [new ExpirationPlugin({ maxEntries: 200, maxAgeSeconds: 30 * 24 * 60 * 60 })] })
);

// App shell
registerRoute(
  ({ request }) => request.destination === 'document',
  new StaleWhileRevalidate({ cacheName: 'app-shell' })
);

// Tree data (SSE endpoint)
registerRoute(
  ({ url }) => url.pathname.startsWith('/api/trees/'),
  new NetworkFirst({ cacheName: 'tree-data', networkTimeoutSeconds: 10, plugins: [new ExpirationPlugin({ maxEntries: 50, maxAgeSeconds: 5 * 60 })] })
);
```

**Decision:** Workbox v7 with CacheFirst for static assets, NetworkFirst for tree data, StaleWhileRevalidate for app shell. All strategies have explicit expiration (max entries + max age).

## 2. IndexedDB vs sql.js WASM for Local Persistence

### IndexedDB (via y-indexeddb)

- **Yjs-native:** `y-indexeddb` is an official Yjs ecosystem provider
- **Zero config:** `new IndexeddbPersistence(docName, ydoc)` — one line
- **Performance:** Writes ~3ms per transaction (IndexedDB overhead). Adequate for CRDT updates (small binary deltas).
- **Storage quota:** Chrome ~60% free disk, Firefox 2GB group limit, Safari ~1GB. Plenty for tree data.
- **No WASM:** Pure JS, no binary payload, instant initialization
- **Maturity:** 920K+ weekly Yjs downloads, used in production (Linear, Notion-like tools)

### SQLite WASM (sql.js / wa-sqlite / official SQLite WASM)

- **OPFS required for persistence:** Synchronous I/O (`createSyncAccessHandle`) only in Web Workers
- **Payload:** 320KB–1.2MB WASM binary (depending on build)
- **Performance:** Sub-millisecond queries for indexed lookups, excellent for structured data
- **Architectural cost:** Requires Web Worker + message passing + WASM startup (~50ms cold)
- **When to use:** Complex queries (JOINs, aggregates, FTS), structured data (Cards), not CRDT sync

### Storage Quota Comparison

| Browser | Storage Mechanism | Typical Limit | Eviction Policy |
|---|---|---|---|
| Chrome | IndexedDB + Cache + OPFS (shared pool) | ~60% free disk | LRU when disk < 20% free |
| Firefox | IndexedDB (group limit) | 2GB max, 10MB min | User prompt on mobile >5MB |
| Safari | IndexedDB | ~1GB | 7-day unused eviction |

### Recommendation

**MVP: IndexedDB only via y-indexeddb.** Yjs's built-in provider handles CRDT persistence. No SQL needed for tree data.

**Post-MVP: SQLite WASM for Cards.** When Cards grow complex (structured queries, FTS on card content), add `wa-sqlite` in a Web Worker. But this is an optimization — the CRDT sync layer stays on IndexedDB.

**sql.js vs wa-sqlite:** wa-sqlite is the correct choice when we add SQLite. It supports OPFS SyncAccessHandle VFS (official SQLite WASM build also does, but wa-sqlite has lighter WASM payload). sql.js is simpler (no OPFS, loads entire DB into memory) but loses data on tab close unless manually persisted.

## 3. CRDT Sync over HTTP/SSE

### Yjs Custom Provider Architecture

```
┌─────────────────────────────────────────────────────┐
│                    Yjs Y.Doc                         │
│  ┌──────────┐  ┌────────────┐  ┌─────────────────┐  │
│  │ y-indexeddb│  │ SSE Provider│  │ HTTP POST sync │  │
│  │ (offline) │  │ (real-time) │  │ (send mutations)│  │
│  └──────────┘  └────────────┘  └─────────────────┘  │
└─────────────────────────────────────────────────────┘
```

### Yjs Custom Provider Template

```typescript
import * as Y from 'yjs';
import { IndexeddbPersistence } from 'y-indexeddb';

class SSESyncProvider {
  private doc: Y.Doc;
  private lastKnownHash: string | null;

  constructor(doc: Y.Doc, treeId: string, baseUrl: string) {
    this.doc = doc;
    this.lastKnownHash = localStorage.getItem(`canopy:tree:${treeId}:hash`);

    // 1. Send local changes to server via HTTP POST
    doc.on('update', (update: Uint8Array, origin: any) => {
      if (origin !== this) { // Don't echo server updates
        fetch(`${baseUrl}/api/trees/${treeId}/updates`, {
          method: 'POST',
          body: update,
          headers: { 'Content-Type': 'application/octet-stream' }
        });
      }
    });

    // 2. Receive remote changes via SSE
    const eventSource = new EventSource(
      `${baseUrl}/api/trees/${treeId}/events?since=${this.lastKnownHash || ''}`
    );

    eventSource.addEventListener('update', (event: MessageEvent) => {
      const update = new Uint8Array(event.data); // binary as base64 or ArrayBuffer
      Y.applyUpdate(doc, update, this); // 'this' as origin prevents echo
      // Store last hash from event metadata
      if (event.lastEventId) {
        this.lastKnownHash = event.lastEventId;
        localStorage.setItem(`canopy:tree:${treeId}:hash`, event.lastEventId);
      }
    });

    eventSource.addEventListener('heartbeat', () => {
      // Keep connection alive, no action needed
    });
  }
}

// Wiring
const ydoc = new Y.Doc();

// Offline persistence
const idbPersistence = new IndexeddbPersistence(`tree:${treeId}`, ydoc);
idbPersistence.on('synced', () => console.log('Loaded from IndexedDB'));

// Real-time sync
const sseProvider = new SSESyncProvider(ydoc, treeId, 'https://canopy.local');
```

### Durable Streams Yjs Protocol (Reference)

The `y-durable-streams` project implements this exact pattern — POST updates, GET via long-poll/SSE. Key design decisions:

- **Binary updates over HTTP:** `Y.encodeStateAsUpdate(doc)` produces `Uint8Array` sent as `application/octet-stream`
- **Last-Event-ID for reconnection:** Server returns SSE `id:` field with each event's hash, client sends `since=<hash>` on reconnect
- **Initial sync:** First connection receives full state via `Y.encodeStateAsUpdate(doc)` from server's canonical copy
- **Conflict resolution:** Yjs handles all merging — server just forwards updates

### Offline → Online Sync Flow

```
1. OFFLINE: User writes messages → Yjs stores in memory + y-indexeddb auto-persists
2. OFFLINE: Mutation queued to IndexedDB outbox (simple list of {treeId, update, timestamp})
3. ONLINE: Service Worker detects connectivity (navigator.onLine + 'online' event)
4. ONLINE: Outbox drained — POST each queued update to server
5. ONLINE: SSE reconnects with Last-Event-ID → server sends missed events
6. ONLINE: Yjs merges local + remote updates → UI reconciles
```

**Decision:** Custom SSE-backed Yjs provider maps 1:1 onto Canopy's SSE transport architecture (T1.1). No WebSocket needed. No y-websocket dependency. HTTP POST for client→server, SSE for server→client, y-indexeddb for offline persistence.

## 4. Background Sync API

### Browser Support (as of 2026)

| Browser | One-off Background Sync | Periodic Background Sync |
|---|---|---|
| Chrome / Edge | ✅ Supported | ✅ Chrome + Android only |
| Firefox | ❌ Disabled by default | ❌ Not implemented |
| Safari | ❌ Not implemented | ❌ Not implemented |

### Strategy: IndexedDB-first, Background Sync as Progressive Enhancement

```typescript
// ALWAYS queue to IndexedDB first
async function queueMutation(treeId: string, update: Uint8Array) {
  const db = await openOutboxDB();
  await db.put('mutations', { treeId, update, timestamp: Date.now() });

  // Try Background Sync (best-effort, browser-dependent)
  if ('serviceWorker' in navigator && 'SyncManager' in window) {
    const registration = await navigator.serviceWorker.ready;
    await registration.sync.register('canopy-sync');
  }

  // Also try immediate POST (if online)
  if (navigator.onLine) {
    drainOutbox();
  }
}

// Service Worker sync event handler
self.addEventListener('sync', (event) => {
  if (event.tag === 'canopy-sync') {
    event.waitUntil(drainOutbox());
  }
});
```

**Decision:** IndexedDB outbox queue is the primary offline mechanism. Background Sync API is a progressive enhancement for Chromium browsers. Firefox/Safari fall back to `navigator.onLine` event listener + periodic drain attempts. The user always sees "2 pending changes" badge sourced from IndexedDB count.

## 5. Delta Chat as Transport Fallback

### Delta Chat Overview
- **Protocol:** Email (IMAP/SMTP) with Autocrypt + PGP encryption
- **Core:** Rust library (`deltachat-core-rust`) with C FFI, Go bindings possible via CGo
- **Latency:** 30 seconds (IMAP IDLE push) to 5 minutes (polling) per message
- **Offline queue:** Built-in — messages composed offline are sent when connectivity returns
- **Metadata:** v2.48+ achieves near-zero metadata leakage to servers

### Suitability Assessment

| Criterion | Delta Chat | Background Sync + HTTP |
|---|---|---|
| Real-time sync | ❌ 30s–5min latency | ✅ <100ms over SSE |
| Offline queueing | ✅ Built-in (email draft) | ✅ IndexedDB outbox |
| Encryption | ✅ PGP end-to-end | ⚠️ MLS post-MVP |
| Multi-device | ⚠️ Email sync across devices | ✅ Yjs CRDT sync |
| Binary data | ❌ Email attachment encoding | ✅ Raw binary over HTTP |
| Deployment | ⚠️ Need email account per user | ✅ Zero additional infra |
| Use case | Last-resort relay when HTTP unavailable | Primary transport |

### Recommendation

**Post-MVP, optional relay.** Delta Chat is the wrong primary transport — 30s+ latency kills real-time collaboration. But as a LAST-RESORT fallback when HTTP is blocked (firewalls, restrictive networks), it's compelling: email is universal, Delta Chat encrypts end-to-end, and the offline queue is built-in.

**Integration point:** The transport adapter interface (T1.8) should define a `DeltaChatAdapter` that wraps `deltachat-core-rust` via CGo. When HTTP/SSE is unavailable and WebRTC fails NAT traversal, fall back to Delta Chat. But this is Phase 3d (post-MVP) work — not needed for single-user local MVP.

## 8. Testing Plan

1. **Offline write → online sync:** Kill network (DevTools offline), write 50 messages, go online, verify all synced
2. **SSE reconnect:** Kill SSE connection for 30s, verify Last-Event-ID replays missed events
3. **Storage quota:** Fill IndexedDB to 80% quota, verify warning banner appears
4. **Background Sync:** Queue mutations, close tab, reopen, verify mutations drained (Chromium only)
5. **Conflict resolution:** Two offline clients write to same node, reconnect, verify Yjs merges correctly
6. **Service Worker update:** Deploy new SW version, verify `skipWaiting()` + `clients.claim()` flow
